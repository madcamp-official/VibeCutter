# D3 / P2 Handoff

## 상태
진행 중 — P2의 D3 선행 조건인 **run-scoped target worktree + Compose overlay build/start**는 이미 구현하고 `26s-w1-c2-04`에서 실제 Docker 기동까지 확인했다. 사용자 결정에 따라 P2는 추가 Dockerize나 취약점 재현을 하지 않는다. 실제 대상 실행, IDOR 재현, evidence 저장과 `verified` 판정은 P3 소유로 넘긴다.

## 변경 파일
- `docs/handoffs/D3-P2.md`: D3 역할 경계와 실제 폐루프의 실행 순서를 기록.

## 제공 인터페이스
- 입력: P1 승인 뒤 생성된 `target_id`와 `run_id`의 target-source Git worktree.
- 출력: `catalog.run_overlay_for(target_id, run_id)`가 생성한 Compose overlay로 해당 worktree를 build context로 사용한 build/start/health 결과, 그리고 `RunScopedTestRunner.run(run_id)`의 대상-worktree regression 결과.
- P3 live 검증용 현재 인스턴스: `26s-w1-c2-04` API `http://127.0.0.1:14017`, UI `http://127.0.0.1:14018`, fixture `.vibecutter/fixtures/26s-w1-c2-04-idor.json`.
- 실패/예외: worktree가 target source Git repository가 아니거나 generated Compose가 isolation 규칙을 위반하면 build/start를 거부한다. P2는 P1 승인 없는 patch apply/reset과 P3 소유 verifier 호출을 수행하지 않는다.

## 검증
- `26s-w1-c2-04` detached target worktree를 build context로 쓴 generated Compose의 `config --quiet`, build, `up -d --wait`, API/UI health와 fixture 재생성을 통과했다.
- 2026-07-18 현재 위 API/UI 컨테이너 모두 healthy 상태로 유지 중이다.
- 최신 main(`1f8835c`)을 P2 branch에 fast-forward했다. P1/P3 D3 handoff가 요구한 `catalog.source_root_for()`와 worktree/overlay 인터페이스가 함께 존재함을 확인했다.
- 아직 실제 target에 대해 P3가 수행하는 `verified`, P1 승인 patch, patch 후 `fixed` 판정은 없다.

## 다음 작업 계획
1. **P3 — live verification:** 현재 `c2-04` 인스턴스와 P2 fixture로 실제 verifier를 실행해 evidence와 `verified` Finding을 만든다. P2는 컨테이너와 fixture를 유지한다.
2. **P1 — patch 승인 배선:** `vc_apply_patch`가 승인된 diff만 `catalog.worktree_manager_for(target_id).create(run_id)`의 target-source worktree에 적용하고, P2 build/restart에 필요한 `target_id`/`run_id`를 전달한다.
3. **P2 — patched runtime 제공:** P1이 승인된 worktree 정보를 전달한 경우에만 해당 run의 overlay build/start/health와 target-worktree regression runner를 실행한다. 취약점 재공격·판정은 하지 않는다.
4. **P3 — post-patch validation:** P2가 제공한 patched base URL을 대상으로 재공격과 정상 기능 검증을 실행하고 evidence를 저장한다.
5. **P1 — judge/fixed 결정:** build·attack·positive·regression·static·scope 게이트를 모아 `fixed` 또는 `RETRY`/`HUMAN_REVIEW`를 결정한다.
6. **인증 Spring 대상은 후속:** `c1-05`의 ephemeral role fixture는 P3가 `bearer_json` replay 및 환경변수 이름 계약을 확정한 뒤에만 P2가 제공한다. token/password는 파일·handoff·evidence에 저장하지 않는다.

## 다른 역할에 필요한 사항
- P1: overlay/worktree 구현은 완료됐다. `vc_apply_patch`의 승인 경계와 P2 호출 입력을 확정해달라. 현재 `c2-04`는 live verifier 증명에는 적합하지만, P3가 지적했듯 무인증 FastAPI라 인증 가드 patch demo의 최종 대상인지 여부는 P1/P3가 합의해야 한다.
- P3: 실제 실행과 보안 verdict를 소유한다. `c2-04` 검증이 끝나기 전까지 현재 컨테이너와 fixture를 유지하므로, 종료·reset이 필요할 때 P2에 알려달라. `c1-05`가 필요하면 먼저 `bearer_json` 계약을 제공해달라.
- P4: 첫 실제 `verified` 또는 `fixed` evidence가 생성된 뒤에만 이를 trajectory/dataset으로 흡수한다. 현재 20개 P2 runtime 체크 결과만으로 학습 성공 사례로 라벨링하지 않는다.

## 결정·가정·리스크
- Dockerized target 수는 20개에서 동결한다. 더 많은 대상보다 evidence가 연결된 실제 trajectory와 최소 1회의 폐루프 재현이 우선이다.
- P2의 책임은 격리된 실행 환경과 재현 가능한 build/reset/regression 기반까지다. P3의 실제 공격 요청과 `verified`/`fixed` 보안 판정을 대행하지 않는다.
- `c2-04`는 P3 live verifier의 즉시 대상이다. patcher/closed-loop 대상은 P3의 인증·code-location 요구에 따라 별도로 확정될 수 있다.
