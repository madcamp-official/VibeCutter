# D4 / P4 Handoff

## 상태

**D4 GPU-free 준비 전량 완료(비교 하네스·벤치마크 음성 샘플·학습/서빙 스크립트). 실학습·실측만 GPU/evidence 대기.**

완료(GPU·실주행 없이 가능한 것):
- **base vs full 비교 하네스**(`eval/compare.py`) — 두 산출물 → metric 델타 표 + 앱별
  개선/악화. **D4 완료기준 "비교표 초안"의 렌더 로직 완성**(두 산출물만 나오면 표가 나옴).
- **벤치마크 음성 정답**(`inventory_benchmark.yaml`) — c2-05(IDOR-clean) 추가. 공개 15개가
  전부 양성이라 못 재던 오탐(precision/FPR)을 이제 측정. **P3 D3-P3 요청 응답.**
- (D2 에서 기록) 모델 서빙 훅(`model/serving.py`) + 7B QLoRA 스크립트(`model/train_lora.py`)
  + `requirements-gpu.txt` + 서빙 런북 — GPU 접근 시 즉시 실행 가능한 상태.

대기(블로커 — 내가 진행할 게 아니라 재료가 나와야 함):
- **7B QLoRA 학습** — GPU 접근(P2) + 실 verified/fixed evidence(P1/P3) → P1 export.
- **base vs full 실비교표** — 위 학습 후 두 산출물. 하네스는 준비 완료.
- **B1/B2 실측** — P2 20개 runtime 실주행 산출물.

근거: 5일 계획 D4 P4 칸("7B QLoRA + OWASP Benchmark + base vs full 비교 / metric·failure
분석", 완료기준 "비교표 초안"), D3-P3(c2-05 음성 샘플 요청), 기획서 12.4/12.5절(metric),
cowork_rule §5(evidence 연결된 것만 학습).

## 변경 파일

- `eval/compare.py`(신규): base vs full 비교.
  - `compare(base, full) -> ComparisonReport`: 두 `BaselineReport` → overall+3군
    metric(P/R/FPR/F1/TPR-FPR) 델타 + 앱별 개선/악화 분류(맞은 양성 − 오탐 기준).
  - `compare_dirs(base_dir, full_dir, truth)`: candidate 디렉토리 두 개 → 두 리포트 → 비교.
  - `ComparisonReport.render()`: `0.67→1.00(+0.33)` 식 델타 표 + 개선/악화 앱 목록.
  - CLI: `python -m eval.compare --base-candidates <dir> --full-candidates <dir>`.
- `eval/test_compare.py`(신규, 4/4): 목 BaselineReport 로 델타·앱분류·음성오탐 검증.
- `datasets/inventory_benchmark.yaml`: **c2-05(CoolAI) IDOR-음성 정답 추가**(expected_vulns: []).
  헤더에 음성 샘플 예외 사유 명시(P3 D3-P3 검증 인용). 벤치마크 15→16개.
- `plan-p4.md`: D4 GPU-free 준비 상태 갱신.

## 제공 인터페이스

```python
from eval.compare import compare, compare_dirs
cmp = compare(base_report, full_report)          # 두 BaselineReport 직접
print(cmp.render())                              # 델타 표
cmp.improved_apps(); cmp.regressed_apps()        # 앱별 변화
# 또는 산출물 디렉토리에서:
cmp = compare_dirs("runs/base/candidates", "runs/full/candidates", truth)
```
음성 샘플: `Inventory.load("datasets/inventory_benchmark.yaml")` → c2-05 의
`expected_vulns=[]` 이 `ground_truth_from_inventory` 에서 빈 정답 집합 → 오탐 측정에 쓰인다.

- `python -m eval.compare` 테스트 4/4. c2-05 음성: 오탐 주입 시 FP=1/precision=0.0,
  flag 없으면 3 TN 확인. 벤치마크 로드 16개.
- 전체 회귀(3.13 venv): eval/model/scanners 14개 모듈 통과.

## 다른 역할에 필요한 사항

**P3**
- c2-05 를 IDOR-음성 정답으로 벤치마크에 반영 완료(D3-P3 요청 응답). 다른 앱의 음성
  판정(완전 clean)이 더 나오면 알려달라 — 같은 방식으로 precision 샘플을 늘린다.

**P1 / P2 (블로커 해제 시)**
- **P2**: GPU 접속 정보 + `nvidia-smi`(7B/양자화 확정). 20개 runtime 실주행 산출물(B1/B2 실측).
- **P1**: 첫 실 verified/fixed evidence 로 run 완주 시 알림 + trajectory export → 그 산출물로
  QLoRA 학습 → base/full 두 산출물을 `compare_dirs` 로 비교표 확정.

**전원**
- 비교표·metric 은 **정답 있는 벤치마크(inventory_benchmark.yaml)에서만** 의미. 학생앱
  (inventory.yaml)은 정답이 없어 precision 대상이 아니다(c2-05 는 P3 검증으로 음성 정답이 된 예외).

## 결정·가정·리스크

- **[결정] 앱별 개선/악화 = (맞은 양성 − 오탐) 증감** — 단순·설명가능. 세밀한 라인단위가
  아니라 (앱×3군) 셀 단위(12.5절 명시할 한계).
- **[결정] c2-05 를 벤치마크에 예외 포함** — 공개 세트가 전부 양성이라 precision 측정 불가.
  음성 정답 1개라도 있어야 "clean 앱에 오탐 안 하는지"를 잰다. 학생 앱이나 이 목적에 한함.
- **[리스크] 음성 샘플이 1개뿐** — FPR 통계력이 약하다. P3 가 추가 clean 앱을 판정하면 보강.
- **[리스크 이월] 학습 데이터 부족** — verified trajectory 얇으면 base 대비 유의차 없음.
  대응: RAG+prompt ablation. 20개 runtime 체크를 학습 성공으로 라벨링 금지(P2/P3 지시).
