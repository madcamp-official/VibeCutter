# D1 / P3 Handoff

## 상태

진행 중. Day1 완료 기준("타깃 1 endpoint ↔ role graph")은 **미달** — 오늘은 P1의 Day1 산출물 검토, P3 모듈 스캐폴딩, verifier 호출 계약 공개까지 했다.

**P1 → P3 블로커는 해소됐다.** Notion 리스크 표가 "P1 evidence store(D2)"를 P3 착수 조건으로 지목했는데, P1이 Day1에 앞당겨 냈고 handoff에 적힌 연동 경로를 실제로 실행해 동작을 확인했다(아래 검증). **이제 P3는 아무에게도 막혀 있지 않다.**

다만 P1 코드를 검증하는 과정에서 **구멍 3개**를 찾았고, 그중 ①은 이 프로젝트의 핵심 주장을 직접 뚫는다. cowork_rule.md 7절("코드 생성만으로 완료 처리하지 않는다")과 Notion 리스크 표의 교차 리뷰 원칙("Infra와 Judge는 서로 다른 사람이 리뷰 — self-confirmation 오류 차단")에 따라 아래에 남긴다.

## 변경 파일

P3 소유 영역만 신규 생성했다. **P1/P2/P4 소유 파일은 한 줄도 건드리지 않았다.**

- `verifiers/types.py`: **P1이 내일 아침 필요로 하는 호출 계약.** `VerifierOutput`(부록 A outputSchema와 동일한 3필드) + `Verifier` Protocol + 부록 A의 `max_requests` 상한 상수. 유일하게 실제 코드가 들어간 파일.
- `verifiers/{access_control,xss,injection}.py`: 취약점군별 verifier (7.3절). docstring 스캐폴딩.
- `surface/{routes,roles,code_index,graph}.py`: Attack Surface Mapper (7.1절). docstring 스캐폴딩.
- `scanners/playwright.py`: 동적 크롤 → `vc_browser_crawl` (7.1절 동적 기반). docstring 스캐폴딩.
- `repair/{locator,patcher,validators}.py`: root-cause locator / repair agent / attack·positive gate 실행기 (7.4~7.6절). docstring 스캐폴딩.
- `.venv/`: 프로젝트 venv 생성 (gitignored).

### 저장소 구조 결정 — 11.2절에서 벗어난 부분

**기획서 11.2절 저장소 구조에는 verifier와 attack surface mapper가 들어갈 자리가 없다.** `scanners/`는 도구별(semgrep/zap/playwright/dependency), `repair/`는 수정 단계별로 나뉘어 있는데, P3의 verifier는 *취약점군별*(access_control/xss/injection)이라 축이 다르다. 그래서 두 디렉터리를 신설했다:

- `surface/` — 7.1절이 "Attack Surface Mapper"라는 독립 엔진으로 정의한 것
- `verifiers/` — 7.3절이 "Exploit Verifier"라는 독립 엔진으로 정의한 것

**`verifiers/`를 `scanners/` 안에 넣지 않은 이유**: `scanners/`는 Candidate를 *생성*하는 쪽이고 verifier는 Candidate를 *판정*하는 쪽이다. 이 프로젝트의 핵심 주장 자체가 "candidate와 verified의 엄격한 분리"(3.2절 가설, 16장 승부처 ②)인데, 둘을 같은 디렉터리에 두면 코드 구조가 그 경계를 흐린다.

`repair/`와 `scanners/playwright.py`는 11.2절 그대로 따랐다. **이견 있으면 말해달라 — 아직 내용물이 없어서 지금이 옮기기 가장 싸다.**

## 제공 인터페이스

### `verifiers/types.py` — P1의 verify tool 배선이 호출할 계약

```python
class VerifierOutput(BaseModel):
    verified: bool
    evidence_ids: list[str]   # 기본값 없음 — verified=false여도 명시적으로 채워야 함 (부록 A required)
    reason: str

class Verifier(Protocol):
    def verify(self, run_id: str, candidate: Candidate, *,
               max_requests: int = 10) -> VerifierOutput: ...
```

- 입력: `run_id`, `Candidate`, `max_requests`(1~20, 부록 A 상한)
- 출력: `VerifierOutput`
- 실패/예외: verifier는 "보안 영향이 있는가"만 판정한다. **policy 검사(`require_target_allowed`/`require_host_allowed`)와 상태 전이는 호출자(P1)가 한다.** verifier는 `mcp_server/`를 import하지 않는다 — 의존 방향은 `mcp_server → verifiers` 한쪽.
- `verifiers/{access_control,xss,injection}.py`가 각각 이 형태의 `verify()`를 노출하고, P1의 tool 본문이 `candidate`를 보고 분기한다.

**P1에게**: `mcp_server/tools_analysis.py`의 `VerifyResult`와 이 `VerifierOutput`이 필드가 완전히 같다. 둘 중 하나는 없애는 게 맞는데, verifier가 mcp_server를 import하면 안 되므로 **`VerifyResult`를 `contracts/schemas.py`로 올리는 것**을 제안한다. 역할 간에 오가는 값이니 공통 계약에 있는 게 맞다. 결정은 P1이 해달라.

## 다른 역할에 필요한 사항

### P1 — 🔴 구멍 ①: 허구 evidence_id로 verified 승격이 된다 (최우선)

`core/state_machine.py:transition_finding()`은 `if not evidence_ids: raise`로 **비어 있는지만** 검사하고, 그 id가 실제로 존재하는지는 보지 않는다. `core/evidence_store.py:update_finding_status()`도 evidence_store 안에 있으면서 Observation 조회를 하지 않는다. 실제로 재현됨:

```python
update_finding_status(fid, FindingStatus.VERIFIED, evidence_ids=["obs-존재하지-않음"])
# → verification_state=verified. 통과.
```

D1-P1.md 리스크 표에 직접 "evidence 없이는 상태 전이 자체가 불가능하도록 하드 가드 **(코드 레벨에서 우회 불가)**"라고 적혀 있는데, 지금은 우회 가능하다. 기획서 16장이 꼽은 MVP 승부처 ②("LLM 주장과 verified finding의 엄격한 분리")와 12.5절 발표 원칙이 전부 이 게이트에 걸려 있다.

**제안**: `update_finding_status()`에서 각 `evidence_id`를 `get(Observation, eid)`로 존재 확인하고, `Observation.run_id == finding.run_id`까지 검사. (`transition_finding()`은 DB를 모르는 순수 함수이므로 그대로 두고, 존재 검사는 store 계층이 맡는 게 맞다고 본다.)

### P1 — 🔴 구멍 ②: secret redaction이 어디에도 없다 (소유자 미정)

cowork_rule.md 4절("secret, token, 개인정보는 evidence, report, dataset 저장 전에 제거한다")과 기획서 4.6절·10.2절(Secret handling)이 요구하지만, `write_artifact()`는 받은 bytes를 그대로 파일에 쓴다. JWT를 넣어보니 artifact에 평문으로 남았다.

**이건 P3에게 특히 위험하다**: 역할 fixture(USER_A/USER_B/ADMIN)로 인증된 트래픽을 주고받는 게 P3뿐이고, IDOR verifier가 하는 일이 정확히 *역할별 `Authorization` 헤더를 바꿔가며 요청/응답을 evidence로 남기는 것*이다. Scrum Helper는 JWT를 쓴다(4.1절). 그 evidence가 D4 밤에 P4의 학습 데이터셋으로 넘어간다. 그리고 **Day4의 P3 완료 기준이 "범위 위반·secret 로그 0 감사"**라 지금 구조로는 자동 실패다.

**제안**: `write_artifact()` 저장 직전에 redaction을 거는 쪽(P1)을 권한다. P4의 SAST 결과나 P2의 빌드 로그에도 secret이 섞일 수 있어 저장 계층 한 곳에서 막는 게 맞다. redaction 규칙 자체(어떤 패턴을 지울지)는 P3가 제공할 수 있다. **소유자를 정해달라.**

### P1 — 🟡 verify tool 본문을 누가 쓰는가 (내일 아침 충돌 지점)

D1-P1.md와 plan.md가 서로 다른 말을 한다:

- D1-P1.md: "각자 **자기 tool의 본문**(현재 `raise NotImplementedError`)만 채우면 된다" → P3가 채움
- plan.md Day2: "verify tool 실배선: MCP tool 호출 → policy_engine 검사 → VERIFYING 전이 → **P3가 구현한 verifier 함수 호출** → evidence_store 기록 → judge 판정" → P1이 채움

둘 다 `mcp_server/tools_analysis.py`의 같은 함수 본문이라 내일 아침 동시에 열면 충돌한다.

**P3 제안: plan.md 쪽(P1이 tool 본문 소유).** cowork_rule.md 2절이 `mcp_server/`를 P1 소유로 못박았고, policy 검사·상태 전이·judge 연동이 전부 P1 영역이다. P3는 `verifiers/`에 순수 함수만 낸다 — 그래서 오늘 `verifiers/types.py`로 시그니처를 먼저 공개했다. **이 전제로 진행하며, 아니면 내일 아침에 말해달라.**

### P1 — 🟡 구멍 ③: `max_requests`에 부록 A 제약이 없다

부록 A는 `{"type":"integer","minimum":1,"maximum":20}`을 명시하는데 실제 생성 스키마는 제약이 없다:

```json
"max_requests": {"default": 10, "title": "Max Requests", "type": "integer"}
```

`max_requests=100000`이 스키마 검증을 통과해 verifier 본문까지 도달한다. 10.2절 "Rate/impact limit" 통제가 비어 있는 셈. 고치려면 `Field(10, ge=1, le=20)`이면 된다.

**위 "tool 본문 소유권"이 정해지기 전이라 P1 파일을 건드리지 않았다** (cowork_rule.md 2절). 소유권이 P3로 정해지면 내가 고치고, P1이면 P1이 고쳐달라. `verifiers/types.py`에 상한 상수(`MAX_REQUESTS_MIN/MAX/DEFAULT`)는 미리 넣어뒀다.

### P1 — 🟡 공통 계약(`contracts/schemas.py`)에 대한 이견

"이견 있으면 지금 이야기해달라"에 대한 답이다. **지금이 가장 싸다** — 마이그레이션 도구가 없고(`SQLModel.create_all`뿐), DB에 데이터가 0건이며, `.vibecutter/`는 gitignored라 아무도 기존 데이터를 안 갖고 있다. Day2에 실제 run이 쌓이면 이 창은 닫힌다.

1. **`Candidate`에 `vuln_class`와 typed 공격 파라미터가 없다** (가장 급함). 현재 `cwe / endpoint(str) / source_symbols / confidence / signals(list[str])`뿐이다. IDOR 하나를 검증하려면 최소한 HTTP method, 소유자 역할, 공격자 역할, 대상 resource 식별자가 필요한데 전부 `signals` 문자열 리스트에 욱여넣게 된다. **P3의 Day2 완료 기준("IDOR verified 1건")이 이 필드에 직접 걸려 있다.**
2. **`Finding.affected_role`이 단수다.** IDOR는 본질적으로 victim/attacker 2개 역할이다.
3. **`Observation.type`이 자유 문자열이다.** P1이 더미 데이터에 `"http_exchange"`를 썼으니 그걸 포함해 값 집합을 고정하자: `http_exchange | db_diff | browser_trace | log | route_map | role_map`. 고정 안 하면 D4 밤 P4의 trajectory 조인이 깨진다. (P4도 확인 필요)
4. **`RootCause`가 `file/symbol/rationale`뿐이다.** 7.4절이 요구하는 근본 원인 점수(dynamic reachability + policy ownership + 최소 수정 범위 + 유사 과거 패치)와 수정 위치 계층(controller hotfix / service policy / shared middleware) 구분을 담을 곳이 없다. Day3에 필요하니 1~3보다는 여유 있다.

### P1 — 🟢 사소한 것

- `mcp_server/resources.py`의 모듈 docstring이 "evidence_store가 아직 없으므로(item 8에서 구현), 오늘은 더미 데이터로 응답한다"인데 evidence_store는 이제 있다. 내용은 여전히 더미가 맞으니(Day2 작업) docstring만 스테일.
- **P3는 Day2에 `vibecutter://findings/{id}` resource를 읽지 않는다** — 아직 더미를 반환하므로 `evidence_store`를 직접 쓴다. Day2에 P1이 실연동하면 그때 전환.

### P2 — role fixture

- manifest 9.3절 `auth.fixtures`(USER_A/USER_B/ADMIN)의 **실물**이 필요하다. IDOR oracle이 "역할 A가 만든 자원을 역할 B가"를 재현하려면 최소 2개의 서로 다른 일반 사용자(USER_A/USER_B)와 각자가 소유한 seed 자원이 있어야 한다 — ADMIN만 있으면 IDOR를 못 만든다.
- 필요한 것: 역할별 인증 획득 방법(로그인 endpoint + 자격증명), 역할별로 미리 소유한 seed 자원의 id, base URL, run 간 reset 방법.
- PDF는 D1 P3 작업에 "role fixture"를, cowork_rule.md 5절은 P2 책임으로 적어 상충한다. **P3가 스펙(어떤 역할·어떤 자원이 필요한가)을, P2가 실물(manifest + DB seed)을** 맡는 것으로 제안한다.
- `policies/scope.yaml`이 비어 있어 지금은 모든 target이 `PolicyViolation`이다. P2가 target 등록해야 verifier가 한 발짝이라도 뗀다.

### P4 — Observation.type / severity 값 집합

- 위 계약 이견 3번(`Observation.type` 값 집합)을 함께 확정하자. P4의 trajectory 조인이 여기 걸린다.
- P3는 SAST를 직접 구현하지 않는다. `vc_run_sast`/`vc_run_sca`/`vc_run_secret_scan`은 D1-P1.md 표에 "P3/P4"로 적혀 있지만 PDF 분업표는 Semgrep 통합을 P4에 배정했다 — **P3는 P4의 candidate를 소비하는 쪽**으로 간주하고 진행한다. 이견 있으면 말해달라.

### 전원 — venv / Python 버전

- `.venv`를 만들어뒀다. 다만 README는 `python3.11`을 지시하는데 이 머신엔 **Python 3.14.3**만 있어 그걸로 만들었다. 기획서 11.1절 요구사항이 "Python 3.11+"이라 충족하고, 전체 스모크 테스트도 통과했다. 다만 팀이 서로 다른 마이너 버전을 쓰면 나중에 미묘한 차이가 날 수 있으니 **팀이 실제로 쓰는 버전을 하나로 정하자** (`datetime.utcnow()`가 3.12+에서 deprecated라 경고가 뜬다).

## 결정·가정·리스크

- **`surface/`와 `verifiers/`를 신설했다** (11.2절에 없음). 이유는 위 "저장소 구조 결정" 참고. 아직 비어 있으니 이견 있으면 지금이 옮기기 가장 싸다.
- **verifier는 `mcp_server/`를 import하지 않는다**고 가정하고 `verifiers/types.py`를 설계했다. P1이 "tool 본문은 P3가 채워라"로 결정하면 이 전제가 깨지므로 내일 아침 확인이 필요하다.
- **P1 소유 파일은 의도적으로 한 줄도 고치지 않았다.** 구멍 ③(`max_requests`)은 5분이면 고치지만 소유권 미정 상태에서 `mcp_server/`를 건드리는 건 cowork_rule.md 2절 위반이다.
- **Day1 완료 기준("endpoint ↔ role graph")에 미달했다.** 오늘은 P1 산출물 검토·계약 공개·스캐폴딩까지였다. Day2 오전에 `surface/routes.py`(소스 기반 route 추출 — P2 target 없이 가능)부터 착수해 만회한다.
- **리스크**: Notion 리스크 표대로 P3의 D3~D4가 전체 병목이다. D3는 P3의 "IDOR closed-loop 완주"와 P1의 "judge 6게이트 완성"이 같은 날 맞물리는데, **`core/judge.py`는 아직 1줄(docstring)이다.** plan.md는 D3 오후 end-to-end 리허설을 예정하고 있으니 그 시간을 미리 잡아야 한다.
- **리스크**: `FindingStatus`와 `RunState`가 VERIFIED/REJECTED/FIXED/HUMAN_REVIEW 이름을 공유한다(P1이 D1-P1.md에서 P3와 재확인이 필요하다고 지목한 항목). **확인했다 — Run은 파이프라인 단계, Finding은 개별 취약점 판정으로 이해하고 진행한다.** verifier는 `FindingStatus`만 다루고 `RunState` 전이는 P1(tool 배선)이 맡는 것으로 가정한다.
