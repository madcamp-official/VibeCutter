# D4 / P3 Handoff

## 상태

두 갈래를 진행했다.

**(1) write-IDOR candidate 생성 경로 완성 — P1 요청 대응, c2-04 라이브 verified.**
D3에 만든 write-IDOR **판정 로직**(`verify_mutation`/`MutationProbe`)은 `MutationProbe`만 받아,
read-IDOR(`verify`가 `Candidate`를 받음)과 시그니처가 비대칭이라 **P1의 tool·dispatch가 read와
같은 `(run_id, candidate)` 패턴으로 못 불렀다**. 오늘 이 비대칭을 없앴다: (a) `candidates_for_target`/
`build_candidates`가 fixture의 `safe_mutation`을 보고 **write-IDOR candidate도 생성**, (b) `Candidate ⇄
MutationProbe` 왕복(`mutation_probe_from_candidate`), (c) dispatch가 `idor_mode`로 read/write를
라우팅. **c2-04에서 candidate 생성→verify_candidate(write)→verified까지 사람 손 없이 라이브 확인**
(evidence 3건). 이제 P1은 read tool 복붙 수준으로 write tool을 얹거나, 기존 `verify_candidate`
단일 진입점을 그대로 쓰면 된다.

**(2) read-IDOR closed-loop을 실제 파이프라인으로 c1-05 라이브 시도 → 갭 3개 실측.**
P1(apply/6게이트)·P2(worktree/overlay)가 D3에 넘긴 "P3가 자기 환경에서 실제 closed-loop 리허설"을
실행했다. **P3 코어(발견→verified→코드위치→패치 합성→worktree 적용 직전)는 전부 라이브로 닫혔으나**,
자동 `FIXED` verdict는 아래 3개에서 멈춘다. **Day3 P0("IDOR closed-loop 완주")는 이 3개가 풀려야
자동으로 초록불이 된다** — 그중 2개는 P3 밖(P1 재조준 배선, 전원 semgrep), 1개는 P3/P1 경로 규약이다.

## 변경 파일

- `surface/candidates.py`: **write-IDOR candidate 생성 경로 추가.** `write_candidate_from_fixture(run_id,
  fixture)`가 fixture의 `safe_mutation`을 `mutation_probe_from_fixture`로 읽어 typed `attack_params`
  (`mutation_method`/`mutation_path`/`observe_path`/`marker_field`/`extra_body`(JSON)/`idor_mode=write`)로
  담은 Candidate를 만든다. `build_candidates`의 `fixture_file` 분기가 read 후보 dedup 후 이 write
  후보를 추가한다. 모듈 docstring에 write 경로 명시.

  > ⚠️ **갱신(main 머지 반영, P2 지적 대응):** 이 handoff 초안은 "`mutation_marker`를 candidate에
  > 담지 않고 재현마다 fresh 생성"하는 내 원안이었으나, 머지 시 P1의 write-tool 커밋(`d6e58de`)과
  > 계약을 맞추며 **P1 계약을 정본으로 채택**했다. **현재 main 확정 계약**: write candidate의
  > `attack_params`에 **`mutation_marker`를 담고**(중첩 `extra_body`는 `extra_body_json` 키에 JSON),
  > P1의 `mutation_probe_from_candidate`가 이 필드를 **필수로 요구**한다. 내 원안 함수
  > (`mutation_probe_from_candidate`/`verify_mutation_candidate`)는 머지에서 제거되고 P1의
  > `mutation_probe_from_candidate`/`verify_mutation_access_control`이 정본으로 남았다.
  > **아래 본문의 "미포함/fresh 생성" 서술은 초안 기준 — 최종 계약은 이 박스가 우선한다.**
- `verifiers/access_control.py`: (초안) Candidate→MutationProbe 왕복 함수 추가. **최종은 위 갱신
  박스대로 P1의 `mutation_probe_from_candidate`(candidate의 `mutation_marker` 필수) + `verify_mutation_access_control` 채택.**
- `verifiers/dispatch.py`: **read/write 분기 추가.** `_idor_verifier`가 `attack_params.idor_mode ==
  "write"`면 `verify_mutation_candidate`로, 아니면 `verify`로 보낸다. `_VERIFIERS["idor"]`를 이
  분기 함수로 교체. `idor_mode` 없는 기존 read 후보는 그대로 read verify(하위호환).

**공통 계약(`contracts/schemas.py`) 무변경** — `Candidate.attack_params: dict[str,str]`에 `idor_mode`/
`mutation_*` 키만 추가(additive). `core`/`mcp_server`/manifest 미변경.

## 제공 인터페이스

### `surface.candidates` — 배치·P1 tool 배선이 호출

```python
# 이제 read + write 후보를 함께 반환(fixture_file + fixture에 safe_mutation 있을 때)
def candidates_for_target(run_id, provisioning, source_root, *, self_signup_hints=None) -> BridgeResult
def write_candidate_from_fixture(run_id, fixture) -> Candidate | None   # safe_mutation 없으면 None
```

### `verifiers.access_control` — write-IDOR을 read와 동일 시그니처로

```python
def mutation_probe_from_candidate(candidate: Candidate) -> MutationProbe   # marker는 매 호출 fresh
def verify_mutation_candidate(run_id, candidate, *, max_requests=10) -> VerifierOutput
```

- 입력: `attack_params`에 `base_url`/`observe_path`/`mutation_method`/`mutation_path`/`marker_field`/
  `extra_body`(JSON str)/`idor_mode=write`/`auth_mode`. `auth_mode`가 `none`이 아니면
  `NotImplementedError`(인증 write 재현 미지원 — 조용히 통과하지 않는다).
- 출력: `VerifierOutput{verified, evidence_ids(3건: before/mutation/after), reason}`.

### `verifiers.dispatch` — 단일 진입점이 read/write 자동 라우팅

```python
def verify_candidate(run_id, candidate, *, max_requests=10) -> VerifierOutput
# vuln_class=idor & attack_params.idor_mode=write → verify_mutation, 아니면 read verify
```

## 다른 역할에 필요한 사항

### P1

- **(write tool 배선 준비됨)** `verify_mutation_candidate(run_id, candidate)`가 read `verify`와 **동일
  시그니처**다. 두 가지 중 택1: (a) `verify_candidate` 단일 진입점을 그대로 쓰면 write 후보가
  `idor_mode`로 자동 라우팅됨, (b) `vc_verify_access_control`에 write mode 옵션/새 tool 추가(read tool
  본문 복붙 수준). policy/승인/상태전이/Finding 판정은 read와 동일 패턴.
- **🔴(closed-loop 갭 1 — 재조준 미배선)** attack/positive 게이트가 **원본 인스턴스**(candidate의
  base_url :14006)를 재타격한다. 패치는 worktree에만 적용되고, 그 worktree를 새 인스턴스로 기동해
  verifier의 base_url을 재조준하는 배선이 없다(`core/judge.py:check_attack` docstring이 "Day3에
  호출부에서 바꾼다"고 예고한 부분). D3-P2 step3-4(patched base_url) 자동화 필요.
- **🟠(closed-loop 갭 2 — apply 경로 접두사)** patcher diff는 `source_root=…/backend` 기준 상대경로
  (`src/main/...`)인데, `worktree_manager`는 **repo root** worktree를 만들어 파일이 `backend/src/...`에
  있어 `git apply`가 실패한다. 소유가 걸친다(patcher=P3, worktree=P2, apply=P1) — **경로 규약을 정하자**
  (아래 결정·리스크 참고). 수동 D3 데모는 backend를 통째 clone해 경로가 맞아 안 걸렸다.
- **(헤드업, 소규모) `VERIFYING→VERIFIED` 미배선**: state_machine 그래프엔 있는데 어떤 wired tool도
  Run을 이 전이로 옮기지 않는다(verify tool은 Finding만 승격). 지금은 오케스트레이터/드라이버가 직접
  전이해야 `vc_generate_patch`(VERIFIED 요구)로 진행된다.

### P2

- **(요청) 다른 타깃에 `safe_mutation` fixture** (c2-04 형식: `resources[*].safe_mutation{method,path,json}`).
  write-IDOR 확장 커버리지가 여기 걸린다 — 없으면 write 후보가 안 생긴다(정상).
- **(closed-loop) c1-05 patched worktree**: overlay build/start + **재조준용 patched base_url**을 P1
  재조준 배선에 넘겨줄 것. 인증 앱 write는 세션/토큰 seed fixture가 있어야 검증 가능(무인증만 현재 지원).

### P4

- **(신규 데이터) write-IDOR verified(c2-04)** — read와 별개 oracle(응답이 아니라 before↔after **상태
  변화**로 판정). trajectory/dataset에 write-IDOR 라벨로 추가 가능. producer는
  `vc_verify_access_control:write_{before,mutation,after}`.
- **🔴(재확인) semgrep Python 3.14 블로커** — static gate가 이것 때문에 안 돈다. 팀 3.11/3.12 통일
  또는 시스템 바이너리 분리 없이는 P1 final judge의 6게이트가 못 채워진다.

## 결정·가정·리스크

- **write oracle은 현재 무인증(auth_mode=none)만.** authed면 `mutation_probe_from_candidate`가
  `NotImplementedError`로 **명확히 거부**한다(조용히 통과 금지). 인증 write 재현(session/bearer로 남의
  자원 변경)은 후속 — P1이 c3-09/c2-08 같은 authed 타깃에 밀려면 이 경로 + P2 seed fixture가 먼저다.
- **write candidate의 marker 처리 = 머지에서 P1 계약 채택**(P2 지적 반영): 초안은 "marker를 candidate에
  안 담고 재현마다 fresh 생성"이었으나, 머지에서 P1의 `mutation_probe_from_candidate`(candidate의
  `mutation_marker`를 필수로 읽음)를 정본으로 채택 → **candidate에 `mutation_marker`를 담는다**. 이 방식은
  P1의 repoint fix(`d6e58de`, 재공격이 fresh 패치 인스턴스를 침)와 함께면 재프로비저닝 충돌이 안 난다.
  (내 원안의 "fresh per replay" 논거는 read-bearer 쪽에 여전히 유효 — 그건 별도 수정 필요.)
- **`observe_path` 유도가 현재 c2-04 형태**(`?owner_id=`, `mutation_probe_from_fixture`가 owner id로
  조립). 다른 앱은 fixture가 `observe_path`를 직접 선언하도록 후속 일반화 필요(P2 계약).
- **apply 경로 접두사 = 자동 배치의 진짜 통합 버그**(수동 데모는 안 걸림). 제안: patcher가 repo-root
  기준 경로로 diff를 내거나(source_dir prefix 부착), apply/worktree가 source_dir로 cwd/`-p`를 맞춘다.
  경로 규약이 정해지면 P3가 patcher 쪽을 맞출 수 있다 — P1과 합의 필요.
- **closed-loop 자동 `FIXED` 미달 = Day3 P0 잔여.** 3개 갭(재조준·apply 경로·semgrep) 해소 시 자동
  초록불. P3 코어 로직(verifier/locator/patcher) 자체는 라이브로 증명됨(WebGoat 전용 아님 재확인).
- **가정**: `verify_mutation`의 안전 경계 유지 — fixture의 `safe_mutation`(되돌릴 수 있는 변경)만 사용,
  파괴적 write(삭제/비번 변경) 금지(10.4절). c2-04 라이브 테스트가 vocab description을 marker로 바꿈
  (reset 커맨드로 원복 가능, 격리 로컬 타깃).
