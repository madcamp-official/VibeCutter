# D1 / P1 Handoff

## 상태

완료. Day1 오전(공통 계약: 데이터 모델 + 상태 머신) + 오후(MCP stdio 서버, resources, evidence_store, policy_engine v1, audit log, tool 24개 스키마+스텁, 실제 stdio Host 연결 검증)까지 계획한 12개 항목 전부 끝났다.

**이 문서를 먼저 공개하는 이유**: cowork_rule.md 5절 "P1은 공통 schema와 tool input/output을 먼저 공개한다"에 따라, P2/P3/P4가 자신의 작업을 이 계약에 맞춰 시작할 수 있도록 오전 중 먼저 게시했고, 저녁에 최종 상태로 갱신한다.

**⚠️ 전원 필독 — 서버 실행 버그 발견/수정**: `python mcp_server/server.py`를 저장소 루트에서 그대로 실행하면 Python이 `mcp_server/` 디렉터리만 `sys.path`에 넣고 저장소 루트는 안 넣어서 `core`/`contracts` import가 전부 깨진다(`ModuleNotFoundError: No module named 'core'`). 실제 MCP Host를 subprocess로 연결하는 검증(item 12)에서 발견했다. `mcp_server/server.py` 상단에 저장소 루트를 `sys.path`에 강제로 넣는 가드를 추가해 고쳤다 — 이제 `python mcp_server/server.py`를 어떤 cwd에서 실행해도(단, cwd 자체는 저장소 루트여야 relative import 외 다른 경로 문제가 없다) 정상 동작한다. MCP Host 설정(Claude Desktop config 등)에 이 서버를 등록할 때는 `command: <repo>/.venv/bin/python`, `args: ["mcp_server/server.py"]`, `cwd: <repo root>` 형태를 쓰면 된다.

## 변경 파일

- `contracts/schemas.py`: 공통 Pydantic 데이터 모델(Target/Run/Observation/Candidate/Finding/Patch/Validation/Trajectory) + 상태 enum(RunState/FindingStatus/ApprovalStatus). 프로젝트 전체가 공유하는 계약.
- `core/state_machine.py`: RunState 전이 그래프(`RUN_TRANSITIONS`) + Finding 상태를 evidence 없이는 전이시키지 못하게 하는 judge-only 게이트(`transition_finding`).
- `core/evidence_store.py`: SQLite(SQLModel) 기반 영속화. 8개 엔티티 저장/조회, artifact SHA-256 hashing, `update_finding_status()`(judge-gate를 저장 계층까지 관통).
- `core/policy_engine.py`: target allowlist(`policies/scope.yaml`) + command_id/typed-args 검증(`policies/commands.yaml`). 미등록 target/command/host는 `PolicyViolation`으로 거부.
- `core/audit_log.py`: `@audited` 데코레이터. tool 이름/args_hash/actor/target/time/result/변경파일을 SQLite에 기록.
- `mcp_server/{resources,tools_inventory,tools_analysis,tools_repair}.py`, `mcp_server/server.py`: MCP stdio 서버 본체. resource 7개 + tool 24개(+vc_ping) 전부 등록, 부록 A 방식 스키마, 전부 `@audited` 적용.
- `policies/scope.yaml`, `policies/commands.yaml`: 실제 정책 파일(현재 빈 allowlist, 형식 예시 주석 포함).
- `core/{judge,planner}.py`: 아직 스캐폴딩만(Day2~4에 채움).

## 제공 인터페이스

### 1. 공통 데이터 모델 (`contracts/schemas.py`)

모든 역할은 이제부터 이 스키마를 통해서만 서로 데이터를 주고받는다. Pydantic v2 `BaseModel` 기반, `python -c "from contracts.schemas import ..."`로 바로 임포트 가능.

| Entity | 주요 필드 | 비고 |
| --- | --- | --- |
| `Target` | id, manifest_hash, source_commit, adapter, allowed_hosts | **`id`는 P2가 만드는 target manifest의 `id`와 정확히 같은 값이어야 한다** (예: `scrum-helper`) |
| `Run` | id, target_id, model_version, tool_versions, status: RunState, started_at/ended_at | |
| `Observation` | id, run_id, type, artifact_uri, hash, producer, timestamp | 모든 evidence의 최소 단위. Finding/Candidate/Validation의 `evidence_ids`가 이 id를 참조 |
| `Candidate` | id, run_id, cwe, endpoint, source_symbols, confidence, signals | 검증 전 가설 (CANDIDATE_SCAN 산출물) |
| `Finding` | id, run_id, title, cwe, owasp_category, severity, verification_state: FindingStatus, evidence_ids, root_cause, patch_ids, validation_id 등 | 부록 B 상세 스키마 채택. `verification_state`는 오직 `core/judge.py`를 통해서만 바뀐다 (아래 참고) |
| `Patch` | id, finding_id, run_id, diff, files, rationale, approval: ApprovalStatus, attempt_no | apply는 `approval=APPROVED` + explicit user confirmation 후에만, git worktree에만 적용 |
| `Validation` | id, run_id, patch_id, build/attack/positive_test/regression/static/scope: bool, verdict | 7.6절 6개 게이트 전부 포함 (11.3절 표엔 scope가 빠져 있어 추가함) |
| `Trajectory` | id, run_id, state, action, result, next_state, reward, label | P4가 Day4에 학습 샘플로 조인할 원자 단위 |

상태 enum:
- `RunState`: `REGISTERED → BUILDING → READY → MAPPING → CANDIDATE_SCAN → VERIFYING → VERIFIED/REJECTED → LOCALIZING → PATCH_PROPOSED → WAITING_APPROVAL → PATCH_APPLIED → VALIDATING → FIXED/RETRY/HUMAN_REVIEW`
- `FindingStatus`: `candidate | verified | rejected | fixed | human_review`
- `ApprovalStatus`: `pending | approved | rejected` (11.3절 표에 없지만 patch 승인 흐름에 필요해 추가)

이 이름들은 고정 계약이다 — 바꾸려면 이 handoff 절차를 다시 거쳐야 한다.

### 2. 상태 머신 (`core/state_machine.py`)

- `transition(current: RunState, target: RunState) -> RunState`: 허용 안 된 전이는 `InvalidTransitionError`.
- `transition_finding(current: FindingStatus, target: FindingStatus, *, evidence_ids: Sequence[str]) -> FindingStatus`: **evidence_ids가 비어 있으면 무조건 `MissingEvidenceError`.** confidence나 서술형 이유를 받는 파라미터가 아예 없다 — LLM 주장만으로는 verified/fixed로 못 올라가게 시그니처 자체로 막아둔 것.
- `is_terminal(state)`, `is_finding_terminal(status)`, `can_transition(...)`, `can_transition_finding(...)`도 제공.

**P3의 verifier/repair 코드는 Finding 상태를 직접 대입하지 말고 반드시 `transition_finding()`을 거쳐야 한다.**

### 3. MCP Tool schema (구현 완료, `mcp_server/`)

기획서 6.6절 카테고리 기준 24개 tool 전부 등록됨. 로직은 P1 소유(register_target/check_readiness 등 일부)를 제외하면 대부분 `NotImplementedError` 스텁이지만, `inputSchema`/`outputSchema`는 전부 최종 형태로 고정되어 있다 — 각자 자기 tool의 본문(현재 `raise NotImplementedError(...)`)만 채우면 된다.

| Tool | 카테고리 | 담당 | 입력(초안) | 출력(초안) |
| --- | --- | --- | --- | --- |
| `vc_register_target` | Inventory | P1/P2 | manifest(dict) | `Target` |
| `vc_inspect_stack` / `vc_check_readiness` | Inventory | P2 | target_id | stack 정보 / readiness bool |
| `vc_build_target` / `vc_start_target` / `vc_reset_target` | Lifecycle | P2 | target_id | `Run` (status 갱신) |
| `vc_map_routes` / `vc_map_roles` / `vc_index_code` | Mapping | P3 | run_id | `Observation[]` |
| `vc_run_sast` / `vc_run_sca` / `vc_run_secret_scan` / `vc_browser_crawl` | Analysis | P3/P4 | run_id | `Candidate[]` |
| `vc_verify_access_control` / `vc_verify_injection` / `vc_verify_xss` | Verification | P3 | run_id, candidate_id | `{verified: bool, evidence_ids: [...], reason}` — 이 결과로 P1의 `transition_finding()`이 `Finding.verification_state`를 갱신 |
| `vc_localize_root_cause` | Repair | P3 | finding_id | `RootCause` |
| `vc_generate_patch` | Repair | P3 | finding_id | `Patch` (approval=PENDING) |
| `vc_apply_patch` | Mutation | P1 게이트, 사용자 승인 필수 | patch_id + 승인 | `Patch` (approval=APPROVED), git worktree에만 적용 |
| `vc_build_and_test` / `vc_replay_attack` / `vc_validate_regression` | Judge | P1 배선(P2/P3 실행기 호출) | patch_id | `Validation` 필드 갱신 |
| `vc_generate_report` / `vc_export_sarif` | Report | P1/P4 | run_id | report artifact URI |

### 4. evidence_store / policy_engine / audit_log 공개 API

- `core.evidence_store`: `save(entity)`, `get(model_cls, id)`, `list_by_run(model_cls, run_id)`, `write_artifact(run_id, *, observation_type, producer, data) -> Observation`(SHA-256 자동 계산), `update_finding_status(finding_id, target_status, *, evidence_ids) -> Finding`(evidence 없으면 예외).
- `core.db`: `get_engine()` — evidence_store/audit_log가 공유하는 단일 SQLite 엔진(`.vibecutter/evidence.db`). 둘 다 각자 엔진을 따로 만들던 걸 오늘 리뷰 중 발견해 통합함.
- `core.policy_engine`: `require_target_allowed(target_id)`, `require_host_allowed(target_id, url_or_host)`, `require_valid_command(command_id, args)`. 전부 실패 시 `PolicyViolation`.
- `core.audit_log`: `@audited` 데코레이터를 tool 함수에 `@mcp.tool()` 바로 아래 붙이면 자동 기록됨. `list_entries(limit)`로 조회 가능.

## 검증

- 실행한 테스트와 결과:
  - `contracts/schemas.py`: 전 모델 인스턴스화 + JSON 직렬화.
  - `core/state_machine.py`: 16개 RunState 그래프 완전성, happy path 시뮬레이션, 잘못된 전이 차단, RETRY 루프, FindingStatus evidence-gating.
  - `core/evidence_store.py`: 8개 엔티티 round-trip, artifact SHA-256 검증, evidence_store를 거친 무증거 승격도 차단됨을 확인.
  - `core/policy_engine.py`: 실제(빈) 정책 파일 기준 미등록 target/command 전부 거부, 임시 fixture로 host/타입/필수인자/미정의인자 세부 로직 검증.
  - `core/audit_log.py`: 성공/스텁오류/승인거부 호출 전부 정확히 기록됨을 확인.
  - **item 12 (실제 Host 연결)**: `mcp.client.stdio`로 실제 subprocess를 띄워 initialize→list_resources→read_resource(5종)→list_tools→call_tool까지 전부 성공. raw stdout 첫 줄을 프로토콜 파서 없이 직접 `json.loads`해서 오염 없음을 확인. **이 과정에서 sys.path 버그를 발견해 수정함(위 경고 참고)** — in-process 테스트만으로는 못 잡는 버그였다.
- 아직 실제 환경에서 검증하지 못한 항목: 없음 (Day1 범위 내에서는 전부 검증됨). Day2부터는 실제 target/run이 생기므로 evidence_store/policy_engine의 진짜 데이터 기준 동작은 다시 확인 필요.

## 다른 역할에 필요한 사항

- **P2**: target manifest(9.3절)의 `id` 필드를 `contracts.schemas.Target.id`와 정확히 같은 문자열로 맞춰달라. `Run.tool_versions`에 P2 adapter가 쓰는 빌드 도구 버전을 넣을 수 있도록 dict 형태(`{tool_name: version}`)로 채워달라. target을 승인할 때는 `policies/scope.yaml`에 `target_id: {allowed_hosts: [...], port: ...}` 형태로 등록해야 policy_engine이 통과시킨다.
- **P3**: `vc_verify_access_control` 등 verifier 구현 시 `core.evidence_store.write_artifact(run_id, observation_type=..., producer=..., data=...)`로 먼저 Observation을 만들고, 그 `.id`를 `core.evidence_store.update_finding_status(finding_id, FindingStatus.VERIFIED, evidence_ids=[obs.id])`에 넘기면 된다 — 이미 구현되어 있으니 Day2 아침에 바로 붙일 수 있다 (Notion 리스크 표 최대 병목 지점 해소). tool 함수를 직접 단위 테스트할 때는 위치 인자로 호출해도 된다 — `@audited` 데코레이터가 위치/키워드 모두 지원하도록 오늘 고쳤다.
- **P4**: inventory 결과를 `Target`으로 채울 때 `manifest_hash`, `adapter`, `allowed_hosts` 필드가 필요하다. `severity`/`owasp_category`는 자유 문자열로 열어뒀으니 SAST 도구(Semgrep) 결과 매핑 시 합의된 값 집합이 있으면 알려달라.
- **전원**: MCP Host 설정에 서버를 등록할 때 `command: <repo>/.venv/bin/python`, `args: ["mcp_server/server.py"]`, `cwd: <repo root>`로 맞춰달라 (sys.path 버그 수정 내용 위 참고). `contracts/schemas.py`의 필드/타입에 이견 있으면 지금 이야기해달라 — 이제부터 이 위에 다른 코드가 계속 쌓인다.

## 결정·가정·리스크

- **Finding은 11.3절 요약 표 대신 부록 B 상세 스키마를 채택**했다 (title/CWE/OWASP/severity/evidence/root_cause/patch/validation 등 포함). 필드명은 11.3절 표 예시(`verification_state`)를 그대로 따랐다.
- **모든 엔티티에 `id`/`created_at`을 추가**했다 (11.3절 표엔 없지만 리소스 URI·audit trail 요구사항상 필수).
- **`Validation`에 `scope` 게이트를 추가**했다 (11.3절 표엔 5개 필드만 있지만 7.6절은 6개 게이트를 요구 — 절대 원칙(패치가 worktree 밖을 건드리면 안 됨)과 직결되는 게이트라 빠뜨릴 수 없다고 판단).
- **`ApprovalStatus` enum을 신규로 추가**했다 (patch generate/apply 분리 + 명시적 승인 흐름을 표현하려면 단순 bool보다 pending 상태가 필요).
- **`RunState.RETRY`는 `PATCH_PROPOSED`로 되돌아가는 것으로 해석**했다 (문서에 명시 안 됨). 재시도 횟수 상한(3회 실패 시 human review)은 상태 머신이 아니라 Day4의 `core/planner.py`가 강제할 예정 — 아직 미구현.
- **리스크**: `FindingStatus`와 `RunState`가 VERIFIED/REJECTED/FIXED/HUMAN_REVIEW 이름을 공유하지만 서로 다른 enum이다. Run은 파이프라인 단계, Finding은 개별 취약점 판정이라는 점을 P3와 재확인 필요 — 헷갈리면 Day2 verify tool 배선 때 버그로 이어질 수 있음.

### Day1 마감 전 셀프 리뷰에서 고친 것

전체 diff를 다시 훑으며 실제로 재현해서 찾은 문제 3가지를 오늘 안에 고쳤다:
- `mcp_server/resources.py`가 `policies/scope.yaml`을 `core/policy_engine.py`와 별도로 읽어서, 파일이 없으면(policy_engine은 `{}`로 방어하는데 resources.py는) `FileNotFoundError`로 죽는 게 재현됨 → `resources.py`가 `policy_engine.load_scope()`를 재사용하도록 통일.
- `core/audit_log.py`의 `@audited`가 감싼 tool 함수를 키워드 전용으로 바꿔버려서, 위치 인자로 직접 호출(예: 단위 테스트)하면 `TypeError: wrapper() takes 0 positional arguments`가 재현됨 → `inspect.signature(fn).bind_partial`로 위치/키워드 모두 지원하도록 수정.
- `evidence_store.py`와 `audit_log.py`가 같은 SQLite 파일에 대해 각자 엔진을 따로 만들고 있어서(중복 코드 + 나중에 동시 쓰기 시 lock 위험) → `core/db.py`로 엔진/경로를 통합.

부수적으로 `write_artifact()`의 `type` 파라미터를 `observation_type`으로 바꿔 builtin shadowing을 없앴고, `VerifyResult.evidence_ids`를 required로 바꿔 부록 A와 정확히 맞췄고, `policy_engine._extract_host()`가 IPv6를 조용히 잘못 파싱하던 걸 명확한 거부로 바꿨다. 전부 재수정 후 전체 스모크 테스트 재실행해서 통과 확인함.
