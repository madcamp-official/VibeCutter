# D1 / P2 Handoff

## 상태
진행 중 — P2 foundation 및 P1 공통 계약 호환(target registration bridge/catalog)을 구현했고, 실제 target 48개 intake와 Compose 격리 사전점검을 마쳤다. 로컬 Docker 런타임 부재로 build/start/health/reset 실행은 차단되어 있다.

## 변경 파일
- `adapters/`: Spring Boot/FastAPI/Node/generic Docker가 공유하는 manifest-driven adapter interface와 registry.
- `runtime/manifest.py`: loopback-only target manifest schema와 YAML loader.
- `runtime/lifecycle.py`: fixed argv command runner, approved reset, health check, test suite result.
- `runtime/worktree.py`: run-scoped detached Git worktree 생성/승인 삭제.
- `runtime/registration.py`, `runtime/catalog.py`, `runtime/readiness.py`: manifest → P1 `Target` 변환, `Target.id` 기반 registry, non-mutating readiness/log location inspection.
- `runtime/test_runner.py`: run ID로만 조회하는 Git worktree regression suite runner.
- `runtime/target_service.py`: P1 policy/evidence contract를 사용하는 target registration/build/start/reset service.
- `runtime/compose_isolation.py`: Docker Compose loopback port/internal network/non-privileged static isolation verifier.
- `mcp_server/tools_inventory.py`: P1이 제공한 schema shell에 P2 inventory/lifecycle 구현을 연결.
- `targets/example-fastapi.yaml`, `targets/README.md`: manifest example과 운영 규칙.
- `tests/`: manifest/lifecycle/worktree security and behavior tests.
- `requirements.txt`, `.env.example`, `.gitignore`: YAML parser와 tracked example env configuration.
- `.vibecutter/targets/sources/` (gitignore된 로컬 작업 경로): `madcamp-official`의 이름에 `w1` 또는 `w2`가 포함된 48개 repository shallow clone. 대상 원본에는 아직 변경하지 않음.
- `docs/handoffs/D1-P2.md`: 실제 target intake 및 실행 전제 점검 결과 기록.

## 제공 인터페이스
- 입력: validated `TargetManifest` (`id`는 P1 `Target.id`와 동일), repository root, fixed `command_id`; reset/remove에는 `approved=True`.
- 출력: `CommandResult(status, exit_code, stdout, stderr, duration_ms)`, `HealthResult(status, attempts, observed_status, reason)`.
- regression 입력/출력: `RunScopedTestRunner.run(run_id)` → `TestRunSummary(status, results, worktree_path)`; test suite가 없으면 `not_configured`이며 통과로 처리하지 않음.
- lifecycle MCP: `vc_register_target(manifest)`, `vc_inspect_stack(target_id)`, `vc_check_readiness(target_id)`, `vc_build_target(target_id)`, `vc_start_target(target_id)`, `vc_reset_target(target_id, approved)`.
- Docker readiness: `docker_isolation` manifest block을 선언한 target은 compose file의 `internal: true` network, service network membership, loopback port binding, `privileged`/`network_mode` 금지를 실행 전에 검사.
- 실패/예외: 외부 base URL, repository escape path, shell syntax, 미등록 command ID, 승인 없는 reset/delete는 거부.

## 다른 역할에 필요한 사항
- P1: target allowlist/registration tool에서 `TargetCatalog(manifest_root=targets/manifests, repository_root=...)`를 초기화한 뒤 `catalog.get(target_id).contract_target`, `catalog.readiness_for(target_id)`, `catalog.lifecycle_for(target_id)`, `catalog.adapter_for(target_id)`, `catalog.test_runner_for(target_id)`를 호출할 연결 지점 필요. MCP caller에는 raw URL/command 대신 target_id와 lifecycle operation만 노출 필요.
- P1: `mcp_server/tools_inventory.py`의 P2 lifecycle body를 연결했다. 실제 target을 추가할 때는 target manifest와 함께 `scope.yaml`의 host/port, `commands.yaml`의 build/start/reset typed command policy를 추가해야 한다.
- P4/target inventory: 실제 Docker Compose target은 manifest `docker_isolation` block과 compose file을 함께 제공해야 readiness가 통과한다.
- P3: candidate replay/regression 시 P2 `test_suites` 결과와 run worktree path를 소비. 패치 apply 전에는 P1 approval gate가 `approved=True`을 전달해야 함.
- P4: 실제 실행 가능한 target repository와 stack·seed·role fixture·health endpoint 정보를 manifest 형태로 제공 필요.
- 사용자/환경: Docker Desktop(WSL 2 backend 포함)을 설치·기동해야 target별 Docker build → loopback health → approved reset을 실제로 진행할 수 있음.

## 결정·가정·리스크
- manifest의 `id`는 공통 `Target.id`와 정확히 일치하며, manifest hash·loopback host:port·adapter는 P1 `Target`으로 변환된다. adapter tool version은 P1 `Run.tool_versions`에 복사한다.
- 공통 MCP schema 변경: `vc_reset_target`에 required `approved: bool`을 추가했다. 이전 `target_id` 단독 호출은 거부되며, destructive reset의 explicit approval 원칙을 구현하기 위한 비호환 변경이다.
- Compose static verifier는 Docker daemon을 실행하지 않는다. 실제 egress 차단과 volume snapshot은 target Compose file·host Docker 환경이 확보된 뒤 별도 smoke test로 검증해야 한다.
- manifest만으로 target이 승인되지는 않으며 P1 scope allowlist와 결합되어야 한다.
- health check는 loopback HTTP만 사용하고 target-controlled redirect도 추적하지 않는다.
- example manifest는 문서용이며 실제 compose file·credential을 포함하지 않는다.
- P2는 `contracts/`, `core/`, `policies/`를 변경하지 않았다. `PyYAML` 추가는 manifest YAML parsing을 위한 runtime dependency다.
- Notion의 repository-level 체크는 clone 또는 README 확인만으로 완료하지 않는다. target별 격리 Compose/manifest, build, loopback health, approved reset이 모두 확인된 뒤에만 체크한다.
