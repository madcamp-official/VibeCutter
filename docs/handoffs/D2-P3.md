# D2 / P3 Handoff

## 상태

Day2 완료 기준("IDOR verified 1건 이상") **달성.** 실제 대상(OWASP WebGoat)에서 IDOR을
자동으로 재현→판정→evidence 저장→Finding `candidate → verified` 승격까지 end-to-end로
확인했다. 이 프로젝트의 MVP 승부처(16장 ②"LLM 주장과 verified finding의 엄격한 분리")를
코드로 처음 증명한 지점이다.

**아직 안 된 것(정직하게):**
- Day1 잔여: role 매핑(`surface/roles.py`) 미완 — role fixture가 있어야 완성된다.
- verifier가 **WebGoat 전용**이다 (2층 로그인 흐름 하드코딩). 실제 몰입캠프 앱엔 아직 안 붙였다.
- **정식 단위 테스트 없음** — 검증은 실제 WebGoat 대상 데모로만 했다(스크래치패드, 커밋 안 함).
- Candidate 스키마 우회 중(아래 참고).

**P2가 실제 target을 올린 것 확인**: `ggultem/concert/codebee/linguaai` isolated runtime이
Day2 사이 main에 들어왔다. **다음 단계는 WebGoat 연습을 이 실제 앱 중 하나에 적용하는 것**이다.

## 변경 파일

- `surface/routes.py`: **신규 구현.** Spring `@*Mapping` 어노테이션에서 (HTTP method, path,
  handler, 소스 위치)를 추출하는 정규식 파서. WebGoat에 돌려 205개 endpoint 추출 확인.
  소스만 있으면 동작(앱 실행 불필요).
- `verifiers/access_control.py`: **신규 구현.** IDOR verifier. 3덩어리 구성:
  `idor_oracle`(판정) + `_replay_idor`(HTTP 재현) + `verify`(조립 + evidence 저장).
- `verifiers/types.py`: (D1에 공개한 계약 그대로, 변경 없음)

## 제공 인터페이스

### `verifiers.access_control.verify()` — P1의 verify tool이 호출

```python
def verify(run_id: str, candidate: Candidate, *, max_requests: int = 10) -> VerifierOutput
```

- 입력: `run_id`, `Candidate`(공격 정보는 현재 `signals`에 key=value로 담김 — 아래 리스크),
  `max_requests`(1~20, 부록 A 상한; 재현에 5회 필요, 초과 시 `ValueError`로 거부 = 10.2절 rate limit).
- 출력: `VerifierOutput{verified, evidence_ids, reason}`. **evidence_ids는 이미 evidence_store에
  저장된 Observation의 실제 id다** — verifier가 `write_artifact`로 먼저 저장하고 그 id를 담는다.
- 부수효과: 요청/응답 2건(baseline, attack)을 `evidence_store.write_artifact`로 저장(redaction 적용).
- **호출자(P1) 몫**: policy 검사, RunState 전이, `update_finding_status(finding_id, VERIFIED,
  evidence_ids=out.evidence_ids)`. verifier는 "보안 영향이 있는가"만 판정한다.

### 재사용 가능한 순수 함수

- `idor_oracle(baseline_body, attack_body, victim_marker) -> (bool, reason)`: "공격 응답에
  피해자 marker가 있고 + baseline엔 없고 + 두 응답이 다르면" verified. WebGoat 없이도 단위 테스트 가능.
- `redact(text) -> str`: JSESSIONID / Bearer 토큰 / password를 `<redacted>`로 치환.

## 검증

실제 WebGoat(Docker, `docker run webgoat/webgoat`)에 verifier를 붙여 확인:

```
verified     : True
reason       : 공격자 세션으로 요청했는데 응답에 피해자 마커 'Buffalo Bill'가 노출됨
               (baseline 응답에는 없음). 수평 권한 통제 부재 (CWE-639).
Finding 상태 : candidate → verified
evidence     : 2건 저장 (baseline=tom 프로필, attack=bill 프로필), 각 SHA-256 hash 포함
redaction    : JSESSIONID=<redacted> 확인
```

- `idor_oracle` 단위: 남의 데이터→True, 같은 데이터→False 확인.
- verifier가 P1 MCP 서버와 동시 import 시 충돌 없음 확인.
- **디버깅 기록(재현한 실패 2건)**: (1) httpx `base_url`이 `/`로 시작하는 경로를 절대경로로
  보고 컨텍스트 경로(/WebGoat)를 버림 → URL 직접 결합으로 수정. (2) WebGoat `UserForm`의
  `@Pattern(regexp="[a-z0-9-]*")` 때문에 언더스코어 든 username은 회원가입 거부 → 데모
  username 형식 수정. 둘 다 소스를 읽어 원인 확정.
- **아직 검증 못 한 것**: 실제 몰입캠프 앱(ggultem 등)에서의 동작. mock 기반 정식 테스트.

## 다른 역할에 필요한 사항

### P1
- **verify tool 본문 배선(소유권 확인 대기 — D1-P3.md 🟡)**: `mcp_server/tools_analysis.py`의
  `vc_verify_access_control` 본문에서 `verifiers.access_control.verify()`를 호출하고, 결과로
  `update_finding_status`를 태우면 된다. 시그니처는 위 "제공 인터페이스" 그대로.
- **D1-P3.md 구멍 ①②③ 여전히 미해결**로 보고 진행 중이다:
  - ① 허구 evidence_id 승격: verifier는 항상 실제 저장한 id만 넘기지만, `update_finding_status`
    자체의 존재 검증이 없으면 다른 호출자가 우회할 수 있다.
  - ② secret redaction: 지금은 **verifier가 임시로** `redact()`를 건다. `write_artifact`
    저장 계층으로 올릴지 소유권 결정 필요. redaction 패턴은 P3가 제공 가능.
  - ③ `max_requests` 부록 A 제약(ge=1, le=20): verifier 내부에선 5>max면 거부하지만,
    tool inputSchema 자체엔 아직 min/max가 없다.
- **Candidate 스키마 개선(D1-P3.md 이견 1)**: 지금 IDOR 공격 정보(base_url, 역할 인증,
  baseline/attack path, victim marker)를 전부 `candidate.signals`에 key=value로 우겨넣고
  파싱한다. `vuln_class` + typed 공격 파라미터 필드가 오면 이 우회를 제거한다.

### P2
- **role fixture(최우선)**: IDOR verifier를 실제 앱에 붙이려면 "역할 A/B 인증 방법 + 각자 소유
  seed 자원"이 필요하다. WebGoat은 레슨 안에 tom/bill이 하드코딩돼 있어 우회했지만, 실제 앱은
  P2가 제공해야 한다. `ggultem` 등 올린 앱 중 **IDOR 검증이 가능한(사용자 2명+각자 자원) 앱
  하나**를 role fixture와 함께 지정해달라.
- verifier가 붙으려면 필요한 것: base URL, 역할별 로그인(endpoint+자격증명), 역할별 seed 자원 id.

### P4
- `Observation.type` 값 집합에 verifier가 쓰는 `http_exchange`를 포함해 확정하자(D1-P3.md 이견 3).
  trajectory 조인이 여기 걸린다.

## 결정·가정·리스크

- **verifier를 3덩어리(oracle/실행/조립)로 분리**했다. `idor_oracle`은 대상 독립적이라 재사용·
  단위 테스트 가능하고, `_replay_idor`만 대상별로 갈아끼우면 된다. WebGoat 전용 부분은
  `_replay_idor`에 격리했고 "P2 role fixture로 대체 예정"으로 주석 표시.
- **httpx를 verifier가 직접 사용**한다. mcp의 전이 의존성으로 이미 설치돼 있지만, verifier가
  직접 쓰니 `requirements.txt`에 명시하는 게 맞다 — P1과 조율 필요(공통 파일이라 조용히 안 바꿈).
- **evidence 저장은 verifier가 직접**(`evidence_store.write_artifact`) 한다. P1 handoff가 제시한
  방식 그대로. verifier는 `core`만 import하고 `mcp_server`는 import하지 않는다(의존 방향 유지).
- **리스크**: 지금 "IDOR verified 1건"은 **개념 증명**이지 일반화된 verifier가 아니다. WebGoat
  전용이고 candidate 우회가 껴 있다. Day3 closed-loop 전에, 실제 앱 1개 + role fixture로
  한 번 더 verified를 만들어 "WebGoat이라서 된 것"이 아님을 확인해야 한다.
- **리스크(일정)**: Day1 role 매핑이 밀렸다. 다만 IDOR verifier에는 WebGoat 레슨의 고정 역할로
  우회 가능했다. 실제 앱으로 가면 role 매핑이 다시 필요하니 P2 fixture와 함께 Day3 초반에 회수.
- **가정**: `FindingStatus`(개별 취약점 판정)와 `RunState`(파이프라인 단계)는 별개 enum이라는
  P1 확인사항을 전제로, verifier는 `FindingStatus`만 다룬다.
