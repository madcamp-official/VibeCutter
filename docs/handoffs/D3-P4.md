# D3 / P4 Handoff

## 상태

**D3 핵심 = HTML/SARIF 리포트 export 완료. 나머지 D3 항목은 실주행 결과 대기.**

완료:
- **HTML/SARIF 리포트 export**(`eval/report_export.py`) — P1 의
  `core.report.build_run_report(run_id) -> RunReport` 를 소비해 finding+evidence+
  patch+validation 을 렌더. **D3 완료기준 "report 1건 export" 충족.**

대기(내가 진행할 게 아니라 남의 실주행 결과가 재료):
- **verified trajectory 축적** — P1 judge 의 실 verified/fixed evidence 필요. 수집
  인프라(`model/trajectory.py`)는 D2 에 이미 완료, 재료만 대기.
- **batch 결과 수집** — P2 20개 runtime 실주행 필요. 수집기(`scanners/batch_scan.py`)는
  이미 완료, 재료만 대기.

근거: 5일 계획 D3 P4 칸("HTML/SARIF 리포트 생성 / verified trajectory 축적 / batch 결과
수집", 완료기준 "report 1건 export"), D3-P1(build_run_report 준비 완료 통보), 기획서
12.2절(리포트).

> 참고: D1·D2-P4 는 완료(inventory·RAG·스캐너·aggregate·baseline·trajectory·vocab).
> D2 GPU-보류 항목(모델 서빙·rerank_fn/embed_fn 훅)은 D2-P4.md 하단에 별도 기록.

## 변경 파일

- `eval/report_export.py`(신규): RunReport → HTML/SARIF.
  - `render_html(report) -> str`: finding 카드(심각도 뱃지·CWE/OWASP·affected_endpoint·
    6게이트 검증 결과·증거 목록·패치 diff). 인라인 CSS, 라이트/다크 대응, 자기완결 HTML.
  - `render_sarif(report) -> dict`: **SARIF 2.1.0**(`$schema` 포함) — GitHub code scanning
    등 표준 도구에 업로드 가능. severity→level(critical/high=error, medium=warning,
    low/info=note), `source_symbols` "파일:줄"→`physicalLocation.region.startLine` 파싱,
    validation 게이트·patch 여부를 `properties` 에 첨부.
  - `export_report(run_id, *, html_path, sarif_path, report=None)`: DB 조립(build_run_report)
    후 파일 쓰기. `report=` 주입 시 DB 우회(테스트/재렌더용).
  - CLI: `python -m eval.report_export --run-id <id> [--html o.html] [--sarif o.sarif]`.
- `eval/test_report_export.py`(신규, 6/6): 목 RunReport 로 렌더/파싱/파일쓰기 검증
  (evidence_store/DB 불필요).
- `plan-p4.md`: D3 리포트 export 상태를 완료로 갱신.

## 제공 인터페이스

```python
from eval.report_export import export_report, render_html, render_sarif
# 실 run:
export_report("run-42", html_path="report.html", sarif_path="report.sarif")
# 데이터 직접 렌더(DB 우회):
html_str = render_html(run_report)
sarif_dict = render_sarif(run_report)   # json.dumps 로 표준 .sarif
```
입력 계약: `core.report.RunReport`(= `FindingReportEntry{finding, evidence[], patch?,
validation?}` 리스트). P1 이 이 형태를 유지하면 렌더러는 수정 불필요.

## 다른 역할에 필요한 사항

**P1**
- `core.report.build_run_report` / `RunReport`(`FindingReportEntry` 필드: finding/evidence/
  patch/validation) **형태 유지 요청** — 렌더러가 이 계약에 직접 붙어 있다. 필드 추가는
  무해, 기존 필드 rename/삭제 시 알려달라.
- MCP tool `vc_generate_report`/`vc_export_sarif` 를 노출한다면 내부에서
  `eval.report_export.export_report(run_id, ...)` 를 호출하면 된다(렌더링은 P4 소유).
- 첫 실 verified/fixed evidence 로 run 이 하나 완주하면 알려달라 — 그 run_id 로 실데이터
  리포트를 한 건 뽑아 D3 "report 1건 export" 를 실 데이터로 재확인한다.

**P3**
- Finding 에 `reproduction_steps`/`impact`/`root_cause` 가 채워지면 HTML 리포트 품질이
  올라간다(현재 비면 해당 섹션 생략). validation 6게이트(build/attack/positive_test/
  regression/static/scope)는 채워지는 대로 뱃지로 표시된다.

## 결정·가정·리스크

- **[결정] SARIF 2.1.0 채택** — 표준 포맷이라 외부 도구 상호운용(GitHub code scanning,
  IDE) 가능. ruleId 는 `Finding.cwe`(없으면 `"vibecutter"`).
- **[가정] severity 는 정규 5단계**(`scanners.vocab.SEVERITY`)로 채워져 온다 — SARIF
  level 매핑이 그걸 전제. 미정규 값이면 기본 `warning`.
- **[결정] 순수 렌더러 / 파일 wrapper 분리** — 실 evidence·DB 없이 렌더러를 완성/검증했다.
  evidence 나오면 `--run-id` 만 주면 실데이터로 나온다(추가 작업 0).
- **[리스크] 실데이터 미검증** — 지금까지는 목 RunReport 로만 검증. 실 run 이 완주하면
  build_run_report 경로(evidence_store 조회)까지 한 번 돌려봐야 완전 확인.
