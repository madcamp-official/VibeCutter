# D3 / P3 Handoff

## 상태

진행 중. **Plan B(IDOR closed-loop)의 P3 로직을 코드로 완성했다** — 재검증(`repair/validators.py`)·
코드 위치(`repair/locator.py`)·패치 생성(`repair/patcher.py`) 전부. 다만 Day3 완료 기준("IDOR
closed-loop 완주")은 **살아있는 대상에서 한 바퀴(패치 apply→재빌드→재기동→재공격)**가 남아 아직
미달이다 — 이 배선은 P2(worktree/build)·P1(apply 게이트)과 실제 role fixture에 걸려 있다.

- 파이프라인: 발견/재현 ✅(D2) → 코드 위치 ✅ → 패치 생성 ✅ → 재공격/정상기능 ✅ (전부 코드+테스트 완료)
- verifier를 `auth_mode`로 일반화해 **WebGoat 전용을 벗어남** + **실제 앱 c2-04(WordNote)에서 live
  IDOR verified 확정**. "WebGoat이라서 된 것" 리스크를 실측으로 닫았다.
- **🎯 c1-05(Scrum Helper, JWT 인증 Spring)에서 closed-loop 한 바퀴 완주** — 발견→verified(bearer)→
  코드위치→패치(v1 500 → v2 house-style 403)→재공격 실패 + 정상기능 통과. 로컬 clone+build+수동
  apply/재빌드로 실증(아래 검증). 기획서 MVP 승부처 ②(증거 기반 verified)·③(패치 closed loop) 실물.
- 남은 것: 최종 `FIXED` 판정(6게이트)은 P1 judge + P2 build/test + P4 static 필요 — P3는 attack/
  positive 2게이트를 실증. apply→재빌드→재기동은 P2 worktree/overlay 자동화 대기(지금은 수동 대행).

**P1의 judge 통합 상황 반영**: P1이 Day3에 `core/judge.py`를 채우며 `check_positive_functionality`가
`repair.validators.validate_patch(run_id, patch_id) -> bool`을 호출하도록 **미리 배선해뒀다**. 그런데
내가 D2 이후 만든 `validate_patch`는 시그니처(candidate 인자)와 반환형(`SecurityValidation`)이 그
계약과 달라, **그대로 두면 judge가 깨지는 상태**였다 — 오늘 최우선으로 P1 계약에 맞춰 고쳤다(아래).

## 변경 파일

- `repair/validators.py`: **신규 구현.** Attack gate + Positive functionality gate 실행기.
  P1 judge용 bool 어댑터(`validate_patch`) + 두 게이트를 재현 1회로 뽑는 엔진
  (`run_security_validation`) + 대상 독립 순수 oracle 2개로 3층 분리(verifier와 동일 구조).
- `repair/locator.py`: **신규 구현.** verified Finding → `RootCause`. 동적(route)·SAST·code_index
  3단 신호로 코드 위치를 짚고, 수정 계층(controller/service/middleware)을 분류한다.
- `repair/patcher.py`: **신규 구현.** `RootCause` → 패치 후보 합성 + 7.5절 랭킹 → 최적
  `Patch(approval=PENDING)`. **generate만, apply 안 함.** 오프라인 템플릿 합성기(Spring IDOR
  가드 삽입) 기본 + LLM 합성기 `synthesize_fn` 훅.
- `verifiers/access_control.py`: **(1) `owner_marker` 선택 필드 추가** — positive gate가 "주인은
  자기 자원을 여전히 본다"를 판정하는 데 쓴다(재현엔 불필요, 기존 동작 무영향). **(2) 재현을
  `auth_mode`로 일반화** — `_replay_idor`를 dispatcher로 리팩터해 `none`(무인증 식별자-스왑, 예
  c2-04)·`session_form`(WebGoat) 두 전략으로 분리(이름·시그니처 유지 → validators 재사용 안 깨짐).
  **(3) `probe_from_fixture`/`candidate_from_fixture` 추가** — P2 IDOR fixture JSON → probe/candidate.
  이제 verifier는 언어(Spring/FastAPI…)를 몰라도 되고 인증 패턴만 분기한다.
  **(4) 공격 파라미터를 `signals` 파싱 → typed `attack_params`로 이전** (D2-P4 최우선 요청). 예전
  signals 파싱은 하위호환 폴백으로만 남김. D1-P3 이견1 / D2-P3 우회 제거.
  **(5) `auth_mode="bearer"` 추가** (JWT 앱, c1-05) — 회원가입 2명 자체 provision + 공격자 토큰으로
  재현(토큰은 메모리 전용, evidence 미기록). baseline/attack_path는 bearer에선 path_template+생성 id로
  런타임 생성이라 optional로 완화. 이제 인증 패턴 3종(none/session_form/bearer).
  **(6) write-IDOR(상태변화 oracle) 추가** — `mutation_idor_oracle`(피해자 자원 before↔after 상태 비교)
  + `verify_mutation` + `MutationProbe` + `mutation_probe_from_fixture`. read-IDOR("봤나")에 더해
  write-IDOR("바꿨나")를 판정(7.3절 "DB/API 상태 비교"). fixture의 `safe_mutation`만 사용(파괴적 변경 금지).
- `repair/patcher.py`: **버그 수정** — `_find_method_span`이 같은 이름의 '호출부'(예: getMyProfile 안의
  `service.getProfile(...)`) 말고 메서드 '선언부'만 잡도록 수정(`) {` / `throws` 판별). c1-05에서 발견.
- `verifiers/dispatch.py`: **신규.** `verify_candidate(run_id, candidate)`가 `candidate.vuln_class`
  (SAST가 채움)로 verifier 모듈을 고른다(idor→access_control, xss/injection→미구현, vuln_class 없으면
  CWE로 보정). D2-P4 "verifier 분기를 vuln_class로" 요청 대응.

## 제공 인터페이스

### `repair.validators` — P1 judge / Plan B 단독 실행이 호출

```python
# P1 judge용 어댑터 (P1이 이미 배선함): positive functionality 결과만 bool로.
def validate_patch(run_id: str, patch_id: str, *, max_requests: int = 10) -> bool

# 두 게이트를 재현 1회로 함께 뽑는 엔진 (judge 없이 P3 단독 실행 시 사용).
def run_security_validation(run_id: str, candidate: Candidate, *, max_requests: int = 10)
    -> SecurityValidation      # .attack, .positive_functionality: GateOutcome / .both_passed

# 네트워크 없는 순수 판정(단위 테스트 가능).
def attack_gate_oracle(baseline_body, attack_body, victim_marker) -> tuple[bool, str]
def positive_gate_oracle(baseline_status, baseline_body, owner_marker) -> tuple[bool, str]
```

- 입력: `validate_patch`는 `patch_id`만 받아 내부에서 `patch → finding → candidate`로 거슬러
  올라가 재현 파라미터(base_url/경로/marker)를 얻는다.
- 출력: `validate_patch` → **bool**(정상 기능 안 깨졌으면 True). `run_security_validation`
  → `SecurityValidation`(두 GateOutcome + `both_passed`). `GateOutcome`은 `{gate, passed, reason,
  evidence_ids}`.
- 부수효과: 패치 후 재현 교환 2건을 `evidence_store.write_artifact`로 저장
  (`observation_type="http_exchange"`, producer `vc_validate_patch:post_patch_{baseline,attack}`).
- 실패/예외: `patch/finding/candidate`가 없으면 `ValueError`. 재현에 필요한 요청 수가
  `max_requests`를 넘으면 `_replay_idor`가 `ValueError`(10.2절 rate limit).
- **판정 원칙**: `Finding.verification_state`를 직접 바꾸지 않는다. 최종 `fixed` 승격은 judge가
  6게이트를 모아 `update_finding_status()`(evidence 실존 검사 포함)로만 한다.

### `repair.locator` — `vc_localize_root_cause`(P1 tool)가 호출

```python
def localize(finding: Finding, *, source_root: str | Path) -> RootCause
```

- 입력: verified `Finding`(`affected_endpoint`/`source_symbols`/`cwe` 사용), 대상 소스 루트.
- 출력: `RootCause(file, symbol, rationale)`. 수정 계층은 `rationale`에 담는다(RootCause에 layer
  필드 없음 — D1-P3.md 이견 4).
- 위치 신호 3단: ① 동적(뚫린 endpoint의 handler를 `surface.routes`로) → ② SAST 교차검증
  (`source_symbols`의 `파일:줄`) → ③ `model.code_index` 폴백 검색.
- 실패/예외: 세 신호 모두 실패하면 `ValueError`(**추측으로 위치를 지어내지 않는다**).

### `repair.patcher` — `vc_generate_patch`(P1 tool)가 호출

```python
def generate_patch(run_id: str, finding: Finding, root_cause: RootCause, *,
                   source_root: str | Path,
                   synthesize_fn: Callable[[Finding, RootCause, Path], list[PatchCandidate]] | None = None,
                   attempt_no: int = 1) -> Patch
```

- 입력: `finding`, locator의 `root_cause`, 대상 `source_root`. 선택 `synthesize_fn`(LLM 대안 후보),
  `attempt_no`(RETRY 회차).
- 출력: `Patch(diff, files, rationale, approval=PENDING, attempt_no)`. **적용하지 않는다** — apply는
  `vc_apply_patch`(P1 게이트 + 사용자 승인 + git worktree)가 별도로 한다(generate/apply 분리, 10.1절).
- 동작: 계층 후보 합성 → 7.5절 랭킹(`security_correctness + regression_safety + architectural_fit
  − size − unrelated − new_dependency_risk`) → 최적 후보를 Patch로. rationale에 랭킹 근거·대안 기록.
- 실패/예외: 후보를 하나도 합성 못 하면 `ValueError`(**빈 패치를 지어내지 않는다**). 오프라인
  템플릿은 Spring `@PathVariable/@RequestParam` handler를 처리 — 그 밖 스택은 LLM `synthesize_fn` 필요.

### `verifiers.access_control` — verify 일반화(auth_mode) 신규 함수

```python
def probe_from_fixture(fixture: dict | str | Path) -> IdorProbe        # P2 fixture JSON → probe
def candidate_from_fixture(run_id, fixture, *, candidate_id=None) -> Candidate  # fixture → Candidate (attack_params 채움)
# IdorProbe.auth_mode: "none" | "session_form" | "bearer"
```

- 재현은 `auth_mode`로 갈린다: `none`(2요청, 무인증 GET 2회)·`session_form`(5요청, 폼로그인+세션)·
  `bearer`(4요청, 회원가입 2명 자체 provision + JWT 헤더; token은 메모리 전용). 상한은 `max_requests`.
- 대상별 endpoint/ID/marker는 코드에 없고 candidate의 **typed `attack_params`**에서 온다(signals는
  하위호환 폴백) → **언어 무관, 인증 패턴만**.

### `verifiers.dispatch` — vuln_class 기반 라우팅 (P1 단일 진입점 옵션)

```python
def verify_candidate(run_id, candidate, *, max_requests=10) -> VerificationResult
def class_of(candidate) -> str | None   # vuln_class 우선, 없으면 CWE로 보정
```

- `candidate.vuln_class`(idor/xss/injection)로 verifier를 고른다. idor→`access_control.verify`,
  xss/injection→`NotImplementedError`(미구현), 판별 불가→`ValueError`.
- P1은 이걸 단일 진입점으로 쓰거나(aggregate.kept 순회) 기존 per-class tool을 유지해도 된다 —
  둘 다 결국 같은 함수를 부른다. verify 대상은 `scanners.aggregate.aggregate(...).kept`.

## 다른 역할에 필요한 사항

### P1

- **(확인) `validate_patch` 계약 충족.** `check_positive_functionality → validate_patch(run_id,
  patch_id) -> bool`(positive만)로 맞췄고 judge가 타고 도는 것 확인했다. 내 재push 후 그쪽 그대로 동작.
- **(배선) `vc_localize_root_cause`**: tool 본문에서 `finding_id → Finding` 조회 + `source_root`(P2
  catalog에서)만 넘겨 `repair.locator.localize(finding, source_root=...)` 호출하면 된다.
- **(배선) `vc_generate_patch`**: `finding_id → Finding`, locator로 `root_cause`, 그리고 `source_root`를
  넘겨 `repair.patcher.generate_patch(run_id, finding, root_cause, source_root=...)` 호출 →
  `Patch(approval=PENDING)`. **`vc_apply_patch`는 P1이 계속 소유** — patcher는 diff만 만들고 적용은
  안 하니, apply 도구가 승인 확인 + worktree 적용 + scope 게이트를 담당해야 한다.
- **(설계 판단) 재공격 2회 재현**: judge가 6게이트를 다 돌리면 `check_attack`(verify 1회) +
  `check_positive_functionality`(내 엔진 1회) = 같은 시퀀스 2회. 원하면 judge가
  `run_security_validation()`을 한 번 불러 attack·positive를 함께 읽어 1회로 줄일 수 있다. P1 판단.
- **(답변) `access_control.redact()` 제거 시점**: 저장계층과 중복이지만 idempotent·안전 →
  Day5 하드닝 때 제거. **(답변) `Candidate.attack_params` 마이그레이션**: 아직 `signals` 파싱 유지,
  patcher 완료 후 이전 예정(강제 아님).
- **(헤드업) `RootCause`가 얇다**: 수정 계층을 `rationale` 문자열에 담는 중. patcher가 계층을
  구조화해 쓰려면 `RootCause` 확장을 그때 논의하자.

### P2

- **(최우선, patcher 전제) 대상 소스 경로 확정**: locator/patcher는 대상 앱 소스가 디스크에
  있어야 동작한다. `.vibecutter/targets/sources/<id>`를 P1 tool이 `source_root`로 넘길 수 있게
  catalog에서 노출해달라.
- **(live loop 마지막 관문) apply→재빌드→재기동 배선**: P3 로직(verifier/locator/patcher/validators)은
  다 됐고, 이제 closed-loop을 실제로 닫으려면 ① patch diff를 대상 repo의 worktree에 적용 →
  ② 재빌드 → ③ 재기동 → ④ 그 인스턴스에 validators 재공격, 이 사이클이 필요하다.
  `WorktreeManager.create(run_id)`가 **VibeCutter repo가 아니라 대상 소스 repo** 기준이어야 패치가
  대상 코드에 적용되고, `RunScopedTestRunner.run(run_id)`도 그 worktree에서 돈다. 이 전제 확인 +
  패치 적용 후 재빌드/재기동을 run별로 돌릴 수 있는지 P1(apply 게이트)과 함께 조율 필요.
- **(해결됨, 감사) IDOR fixture 제공**: `26s-w1-c2-04`에 사용자 2명 + 각자 자원 fixture를 줬다.
  verifier를 여기에 붙게 일반화 완료.
- **(신규 계약) fixture 필드 이름은 이제 P2↔P3 계약**: `probe_from_fixture`가 fixture의
  `resources[*].read_path`+`victim_marker`(피해자), `baseline_path`+`marker`(공격자), 최상위
  `base_url`, `authentication.mode`에 의존한다. **다른 앱 fixture도 이 필드 이름을 유지**해달라.
- **(요청) c2-04 live 인스턴스**: verifier 실검증을 위해 c2-04를 띄운 상태(:14017)에서
  `prepare_idor_fixture`를 돌려 `.vibecutter/fixtures/26s-w1-c2-04-idor.json`를 만들어두거나,
  P3가 로컬에서 재현할 수 있게 소스 clone 경로/기동 방법을 확인해달라.

### P4

- **(완료) D2-P4 최우선 요청 반영**: verifier 분기를 `vuln_class`로(`verifiers/dispatch.py`),
  공격 파라미터를 `signals`→typed `attack_params`로 이전. verify 대상은 `aggregate.kept`(원소가
  Candidate라 verifier 코드 변경 없이 소비). **주의**: SAST 후보는 `vuln_class`는 있어도
  `attack_params`(base_url·경로·marker)는 없다 → 실제 verify 전에 fixture의 replay 파라미터를 합쳐야
  한다(c2-04는 `candidate_from_fixture`가 합본 생성).
- **(FYI) locator가 `model.code_index`를 소비**한다(route 폴백 시 `CodeIndex.build(root).search(q,
  k=1)` → `hit.chunk.file`). D1-P4에서 예고한 소비 지점이 실제로 붙었다. API 안정 유지 부탁.
- **(확인) SAST `source_symbols` 포맷**: locator가 `"파일:줄"` 형식으로 교차검증한다(semgrep_runner
  출력 그대로). 이 포맷 유지되면 됨.
- **(FYI) 새 validation evidence**: 패치 후 재현을 `http_exchange`로 저장(producer
  `vc_validate_patch:post_patch_{baseline,attack}`). trajectory 조인·리포트에 validation 근거로 사용 가능.
- **(신규) c2-05 = IDOR-음성(clean) 실제 앱 라벨**: verified 0이 정답인 ground-truth 음성 샘플이다.
  ① precision 측정(오탐 없음 확인)의 기준선, ② 모델의 "안 뚫리는 코드는 flag 안 하기" 학습용
  negative sample로 쓸 수 있다. `inventory_benchmark.yaml`에 실제 앱 음성 라벨로 추가 검토 요망
  (지금까지 확정: c2-04=IDOR 양성/무인증, c1-05=IDOR 양성/JWT, **c2-05=IDOR 음성**).

## 결정·가정·리스크

- **`validate_patch`를 P1 계약(bool, positive만)에 맞춰 조정**했다. 원래 만든 "attack+positive를
  재현 1회로 함께 뽑는" 로직은 버리지 않고 `run_security_validation()`으로 남겨, Plan B 단독 실행과
  어댑터가 공유한다. `mcp_server`/`core`는 건드리지 않았다.
- **positive gate는 지금 baseline(공격자의 자기 자원 접근)만 확인**한다. "피해자도 자기 자원을
  여전히 본다"까지 보려면 두 번째 역할 재인증이 필요하고, 이는 P2 role fixture가 붙으면 추가한다.
  강한 positive 판정을 하려면 candidate에 `owner_marker`가 있어야 한다(선택 필드).
- **locator는 근본 원인을 추측으로 지어내지 않는다** — route/SAST/code_index 모두 실패하면
  `ValueError`. (verified만 승격하는 프로젝트 원칙을 위치 추적에도 적용.)
- **patcher는 generate만, apply 안 한다** — `Patch(approval=PENDING)`만 반환하고 원본/worktree를
  건드리지 않는다(generate/apply 분리, 10.1절). 랭킹은 결정적이고 diff 합성만 pluggable하다:
  오프라인 템플릿(Spring 가드) 기본 + LLM `synthesize_fn` 훅. **패치가 완벽할 필요는 없다** —
  build/attack/positive/regression 게이트가 걸러내고 실패 시 `attempt_no`를 올려 재시도(planner의
  3회 실패 → HUMAN_REVIEW)하는 구조라, patcher는 "가장 그럴듯한 후보"를 낼 뿐이다.
- **(c1-05 실증) 템플릿 patch의 한계 = LLM 합성기의 근거**: 오프라인 템플릿 v1은 표준
  `ResponseStatusException`을 써서 **공격은 막았지만(보안 OK) 이 앱의 `GlobalExceptionHandler`에
  걸려 500** 반환. 앱 규약 `BusinessException(ErrorCode.FORBIDDEN)`을 쓴 v2라야 깔끔한 403.
  → 템플릿은 "보안은 맞는 빠른 초안", house-style 패치는 코드 문맥을 읽어야 나옴(= `synthesize_fn`
  LLM 합성기 몫). 다중 게이트+재시도 설계가 정확히 이런 500을 걸러 v2로 가게 하는 장치다.
- **(스코프) apply→재빌드는 수동 대행**: c1-05 closed-loop에서 patch 적용·재빌드·재기동을 disposable
  clone에서 수동으로 했다(검증 후 소스 원복). 실제 파이프라인은 P2 run-scoped worktree + compose
  overlay가 자동화한다. 원본 branch/canonical 소스는 건드리지 않았다.
- **(음성 데이터의 가치) c2-05는 "잘 잠긴 앱"** — IDOR 표면이 없어 verified가 0인 게 정답이다. 빈
  결과가 아니라: ① **오탐 저항(verified precision, 12.4절)의 근거** — 안 뚫리는 앱에서 verified를
  만들지 않음을 실증. ② **attack surface mapper 스펙 확정** — 취약 신호 = *id-addressable + 교차
  사용자 + current_user 미스코프* (c1-05 `/{userId}/profile` vs c2-05 `/users/me`의 차이). 지금 stub인
  `surface/roles.py`·`graph.py`가 이 신호로 자동 분류해야 한다. ③ **모델 음성 샘플**(아래 P4).
- **(패턴) 실제 앱 IDOR은 hit-or-miss**: c2-04(무인증→다수)·c1-05(방어 잘했지만 프로필 1건)·
  c2-05(완전 clean, 0건). "clean"도 정직히 기록하는 게 precision 신뢰의 핵심.
- **(신규 축) write-IDOR = 상태변화 oracle**: read는 응답 body에서 marker를 찾지만, write는 피해자
  자원을 before/after로 **독립 관찰**해 실제 변경을 확인한다("200 떴다"가 아니라 상태 비교, 7.3절).
  c2-04에서 실증. c2-04는 무인증이라 verifier-side 확장이고(패치는 여전히 구조 변경), oracle 자체는
  대상 독립·재사용. 변경은 `safe_mutation`(되돌릴 수 있는 것)만 — 파괴적 write(계정 삭제/비번 변경) 금지.
- **(발견) `RootCause`에 layer 필드 없음** → patcher가 계층을 `RootCause.symbol/file`로 재분류
  (`locator._classify_layer` 재사용). 계층 정보를 구조화해 넘기려면 `RootCause` 확장 필요(P1 협의).
- **(해소) D1-P3 이견1 / D2-P3 signals 우회 제거**: P1의 `Candidate.attack_params`(typed) + P4의
  SAST `vuln_class` 채움으로, 공격 파라미터/verifier 분기를 문자열 signals 대신 typed 필드로 이전했다.
  기존 signals 후보는 폴백으로 계속 동작(하위호환).
- **(발견) `surface/routes.py` 소규모 버그**: 클래스 레벨 `@RequestMapping("/path")`를 endpoint
  route로도 중복 추출한다(`/path/path` 형태). locator는 정확 매칭+method 필터라 이번엔 영향
  없었지만, 후보 매칭 정확도에 영향 가능 — routes.py 정리 시 함께 고칠 것.
- **리스크(WebGoat 전용) — 해소 완료**: verifier를 `auth_mode`로 일반화해 c2-04(무인증 FastAPI)에서
  **실제 live IDOR verified를 확정**했다(로컬 clone+up). verifier는 언어에 의존하지 않으므로(HTTP만),
  앱이 늘어도 인증 패턴만 추가하면 된다.
- **패치 데모 대상 분리 필요**: c2-04는 verifier 증명엔 최적(깨끗한 ID IDOR)이지만 **인증이 없어
  patcher/closed-loop 데모엔 부적합**하다. 인증 있는 Spring IDOR 타깃(예: c1-05)에 fixture를 하나 더
  받으면 Spring routes/patcher 템플릿이 그대로 붙어 closed-loop 패치 데모가 깔끔해진다(P2 요청 고려).
- **가정**: `FindingStatus`(취약점 판정)와 `RunState`(파이프라인 단계)는 별개 enum. validators는
  Finding 상태를 직접 바꾸지 않고 게이트 bool만 낸다.
