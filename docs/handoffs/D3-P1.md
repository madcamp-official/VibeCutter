# D3 / P1 Handoff

## 상태
진행 중 — Day3 작업 시작. D1-P2.md(12:14 갱신본)와 D2-P4.md를 읽고 오늘 실행 순서를 확정했다
(자세한 순서는 `plan.md` Day3 섹션 "D1-P2.md + D2-P4.md 반영 — 오늘 실행 순서" 참고). 순서상
가장 먼저 P2에게 필요한 요청 하나를 병행 착수를 위해 먼저 공유하고, 나머지 항목은 진행하면서
이 파일에 계속 추가한다.

  `.vibecutter/trajectories/<run_id>.jsonl`에 상태 전이 스텝을 기록(D2-P4.md 요청 (a) 결정).

  `_prepare_scan()` 공통 헬퍼로 policy 검사 + CANDIDATE_SCAN 전이(멱등)를 처리.

  쓴다(전엔 `_REPO_ROOT / manifest.source_dir` 수작업 조합). 사실이 아니게 된 "3개 manifest

- `tests/test_scan_tool_wiring.py`(신규), `tests/test_localize_root_cause.py`: 위 변경 회귀 테스트.

  (D3-P3.md 요청). `root_cause`는 `vc_localize_root_cause`와 별도로 이 tool 안에서

  하나도 합성 못 하면(`ValueError`) run 상태를 바꾸지 않는다.
- `tests/test_generate_patch.py`(신규): 위 배선 회귀 테스트 4건.
- `core/judge.py`: **`check_scope` 게이트 실제 구현** + `diff_touched_files()`/`assert_diff_within_worktree()`/
  `ScopeViolationError` 신규 공개 — unified diff의 `+++ b/<path>` 헤더에서 건드리는 파일을
  뽑아 worktree 경로 밖이면 거부한다(10.1절 절대 원칙). `core`가 `mcp_server`에 의존하지
  않도록 `_service()`를 `runtime.target_service`에서 직접 재구성(레이어링 유지).
- `mcp_server/tools_repair.py`: **`vc_apply_patch` 실제 구현** — `catalog.worktree_manager_for()`로
  run-scoped worktree 생성(이미 있으면 재사용) → `assert_diff_within_worktree()`로 사전
  scope 강제(judge의 사후 검증과 동일 규칙 공유) → `git apply`로 적용. RunState는
  PATCH_PROPOSED→WAITING_APPROVAL→PATCH_APPLIED 전이. 이미 PATCH_APPLIED인 run으로
  재호출되면(재시도) diff를 다시 적용하지 않고 그대로 반환 — 두 번째 `git apply`가
  "already applied" 에러를 내는 것을 방지.
- `tests/test_apply_patch.py`(신규): 실제 임시 Git repo에서 `git worktree add` + `git apply`
  end-to-end 검증(원본 branch 미변경 확인 포함) 5건.
- `core/judge.py`: **`check_build`/`check_regression`/`check_static` 게이트 실제 구현 + `compute_verdict()`
  신규.** `check_build`/`check_static`은 P2 `RunScopedTestRunner`가 test suite에 쓰는 것과 같은
  패턴(`manifest.model_copy(update={"source_dir": "."})`)으로 worktree를 build/scan 대상으로
  삼는다. `check_regression`은 `catalog.test_runner_for(target_id).run(run_id)`를 그대로
  호출(이미 worktree 전용이라 블로커 없음). `check_static`은 원본 source와 patched worktree
  양쪽에 P4 semgrep을 돌려 high/critical 후보 수를 비교한다. `compute_verdict(validation)`은
  6게이트가 전부 채워졌을 때만 `FIXED`/`RETRY`를 반환(재시도 횟수 상한은 Day4 planner 소관).
  **알려진 한계**: `check_build`는 build command가 `working_dir`로 P2 static Compose overlay를
  가리키는 target(현재 checked-in 대부분)에서는 아직 patched worktree가 아니라 원본 source를
  빌드한다 — P2의 run-scoped overlay가 오면 코드 변경 없이 patched build를 검증하게 된다.
- `core/evidence_store.py`: `find_or_create_validation(run_id, patch_id)` 추가(patch당 Validation
  row 지연 생성, `find_or_create_finding`과 같은 패턴).
- `mcp_server/tools_repair.py`: **`vc_build_and_test`/`vc_replay_attack`/`vc_validate_regression`
  실제 배선.** 세 tool이 patch당 하나의 Validation row를 공유(각자 맡은 게이트만 채움).
  `_advance_to_validating()`으로 PATCH_APPLIED→VALIDATING 전이(멱등), `_finalize_validation()`이
  6게이트가 모두 채워지면 `compute_verdict()`로 RunState를 FIXED/RETRY로 전이하고, FIXED면
  validation 요약을 evidence artifact로 남겨 `update_finding_status(FIXED, evidence_ids=[...])`
  호출(하드 가드 유지), RETRY면 Finding은 그대로 둔다.
- `tests/test_judge.py`: check_build/check_regression/check_static/compute_verdict 회귀 테스트 10건.
- `tests/test_validate_patch_tools.py`(신규): 3-tool 연쇄 호출로 FIXED/RETRY 양쪽 verdict 경로,
  선행 조건 위반, 존재하지 않는 patch 거부 4건.
- `core/report.py`(신규): `build_run_report(run_id)` — finding+evidence+patch+validation을
  run 단위로 조인(D2-P4.md 요청 (c) 응답). P4의 HTML/SARIF export가 그대로 순회 소비할 입력
  데이터 소스. `vc_generate_report`/`vc_export_sarif`는 이 데이터 소스를 언급하도록
  docstring만 갱신했고, 실제 렌더링(P4 소유)은 아직 배선하지 않았다.
- `tests/test_report.py`(신규): 조인 정확성(evidence/patch/validation 매칭), patch 없는
  finding도 보고되는지, 존재하지 않는 run은 빈 report인지 3건.

## 제공 인터페이스
- `core.report.build_run_report(run_id) -> RunReport` — `RunReport.findings: list[FindingReportEntry]`,
  각 entry는 `finding`(Finding), `evidence`(Observation 리스트), `patch`(Patch|None),
  `validation`(Validation|None). P4가 HTML/SARIF export에서 그대로 소비하면 된다.

## 다른 역할에 필요한 사항
- **P2 (선통보, 병행 착수 요청)**: patch가 실제로 나오면(오늘 후반 `vc_apply_patch`가
  `catalog.worktree_manager_for(target_id).create(run_id)`로 만드는 run-scoped worktree) 그
  worktree를 build context로 쓰는 run-scoped Compose overlay가 필요하다. D1-P2.md도 이걸
  다음 단계로 예고했다. 오늘 judge의 build/regression gate(`check_build`/`check_regression`)를
  실 target으로 검증하려면 이게 있어야 하므로, 내가 나머지 배선(`vc_run_sast`/`vc_run_sca`,
  `vc_generate_patch`, `vc_apply_patch`, judge 게이트)을 진행하는 동안 병행 준비를 부탁한다.
  우선 대상은 `26s-w1-c2-04`(오늘 첫 closed-loop 리허설 target)로 맞춰주면 좋겠다.
