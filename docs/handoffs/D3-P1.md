## 제공 인터페이스
- `vc_run_sast(run_id)`/`vc_run_sca(run_id) -> ScanResult`: policy 검사 + CANDIDATE_SCAN 전이
  + `scanners.sast.run_semgrep`/`scanners.sca.run_osv` 호출 + `aggregate(...).kept` 저장.
- `vc_localize_root_cause(finding_id) -> RootCause`: `catalog.source_root_for(target_id)` 기반.
- `vc_generate_patch(finding_id) -> Patch`: `Patch(approval=PENDING)`, 실패 시 run 상태 불변.
- `vc_apply_patch(patch_id, confirmed) -> Patch`: `confirmed=True` 없이 거부, 성공 시
  `Patch.approval=APPROVED` + Run `PATCH_APPLIED`.
- `vc_build_and_test(patch_id)`/`vc_replay_attack(patch_id)`/`vc_validate_regression(patch_id) -> Validation`:
  patch당 공유 Validation row, 6게이트 전부 채워지면 `verdict`(`FIXED`/`RETRY`) 확정 + Finding/Run 전이.
- `core.judge`: `check_build`/`check_regression`/`check_static`/`check_scope`(신규 구현),
  `compute_verdict(validation) -> "FIXED"|"RETRY"|None`, `diff_touched_files`/`assert_diff_within_worktree`/
  `ScopeViolationError`(scope 검사 공유 유틸).
- `core.trajectory.record_trajectory_step(...)`: run별 `.vibecutter/trajectories/<run_id>.jsonl`.
- `core.report.build_run_report(run_id) -> RunReport` — `RunReport.findings: list[FindingReportEntry]`,
  각 entry는 `finding`(Finding), `evidence`(Observation 리스트), `patch`(Patch|None),
  `validation`(Validation|None). P4가 HTML/SARIF export에서 그대로 소비하면 된다.

## 다른 역할에 필요한 사항
- **P2 (선통보, 병행 착수 요청)**: patch가 실제로 나오면(오늘 후반 `vc_apply_patch`가
  `catalog.worktree_manager_for(target_id).create(run_id)`로 만드는 run-scoped worktree) 그
  worktree를 build context로 쓰는 run-scoped Compose overlay가 필요하다. D1-P2.md도 이걸
  다음 단계로 예고했다. 오늘 judge의 build/regression gate(`check_build`/`check_regression`)를
  실 target으로 검증하려면 이게 있어야 하므로, 내가 나머지 배선(`vc_run_sast`/`vc_run_sca`,
  `vc_generate_patch`, `vc_apply_patch`, judge 게이트)을 진행하는 동안 병행 준비를 부탁한다.
  우선 대상은 `26s-w1-c2-04`(오늘 첫 closed-loop 리허설 target)로 맞춰주면 좋겠다.
- **P2 (신규)**: `26s-w1-c2-04` `check_readiness()`가 `unavailable_executables=['py']`를
  보고한다 — manifest 커맨드가 Windows `py -3.13` launcher를 참조하는데, macOS/Linux에는
  `py`가 없다(보통 `python3`/`python`). D1-P2.md 작업이 Windows 환경 기준이라 생긴 이식성
  격차로 보인다. 데모/통합 환경이 여러 OS에 걸치면(D5 클린 환경 재현 항목과도 연결) 이 부분도
  같이 정리가 필요할 것 같다.
- **P3**: Repair/Mutation/Judge 전 구간 배선 완료 — `vc_localize_root_cause`(catalog.source_root_for
  사용으로 정리), `vc_generate_patch`(patcher.generate_patch 호출, root_cause는 이 tool 안에서
  직접 재계산), `vc_apply_patch`(worktree 생성+scope 강제+git apply, confirmed 게이트),
  `vc_build_and_test`/`vc_replay_attack`/`vc_validate_regression`(6게이트 전부 실제 구현,

- **P4**: report용 데이터 조인(`core.report.build_run_report(run_id) -> RunReport`) 준비 완료

## 결정·가정·리스크
- **[결정] trajectory 기록 주체는 P1**: 상태 전이 label(verified/fixed/rejected/human_review)이
  P1 judge 판정 순간에만 확정되므로, P4가 evidence_store에서 사후 조립하지 않고 P1이 각
  tool 호출 지점에서 `core.trajectory.record_trajectory_step()`을 직접 호출한다(D2-P4.md
  요청 (a) 응답 — P4 자신의 문서에도 "P1이 호출하거나(권장)"이라고 돼 있어 이미 P4가
  기대하던 방향과 일치).
- **[결정] severity/owasp vocab 채택**: `scanners.vocab.SEVERITY`/`OWASP_2021` 값 집합을
  그대로 채택해 `find_or_create_finding()`에서 `Finding.severity`/`owasp_category`를 채운다
  (D2-P4.md 요청 (d)). 스키마 필드는 이미 있었고 지금까지 비어 있었을 뿐이라 additive,
  마이그레이션 걱정 없음.
- **[결정] `aggregate.kept`는 스캐너별 개별 호출**: `vc_run_sast`/`vc_run_sca`가 각자 자기
  스캐너 결과만 `aggregate()`에 넣어 FP reject+우선순위를 적용한다(D2-P4.md 요청 (b)).
  **알려진 한계**: SAST·SCA 교차 중복 제거는 안 된다 — 두 tool이 독립 호출되기 때문. "스캔
  완료" 시점을 묶는 별도 단계(예: planner, Day4)가 생기면 그때 cross-scanner aggregate로
  바꾼다.
- **[결정] `check_attack`+`check_positive_functionality` 통합 보류**: P3가 제안한
  `run_security_validation()` 통합(재공격 2회→1회)은 이번엔 하지 않았다. `check_attack`은
  이미 Day2에 실 target(WebGoat)으로 검증된 코드라 지금 리팩터링하면 회귀 리스크가 생기고,
  통합의 이득은 정확성이 아니라 효율(HTTP 재현 1회 절약)이라 우선순위가 낮다고 판단했다.
  Day4/5 하드닝 때 재검토.
- **[결정] RETRY는 Finding을 건드리지 않는다**: judge verdict가 `RETRY`면 Run만 전이하고
  Finding은 `VERIFIED`로 그대로 둔다 — `FINDING_TRANSITIONS` 그래프 자체가 `VERIFIED`→
  `{FIXED, HUMAN_REVIEW}`만 허용해 `RETRY`에 대응하는 Finding 상태가 없다. 3회 연속 실패 시
  `HUMAN_REVIEW`로 보내는 재시도 횟수 상한은 이 코드가 아니라 `core/planner.py`(Day4)가
  강제하기로 한 기존 설계(`core/state_machine.py` 주석)를 그대로 따른 것 — 오늘 새로
  만든 결정이 아니라 확인만 했다.
- **[알려진 한계] `check_build`는 Compose 기반 target에서 아직 "패치된" 빌드를 증명 못 함**:
  P2 `RunScopedTestRunner`와 같은 `source_dir="."` 치환 패턴을 썼지만, checked-in manifest
  다수의 build command는 `working_dir`로 P2 root의 static Compose overlay를 가리키고 그
  overlay의 build context가 아직 원본 source clone을 고정 참조한다(D1-P2.md가 예고한 다음
  작업). `working_dir` override가 없는 순수 source-native build(Gradle/npm 등)에서만 지금
  이 게이트가 실질적으로 patched 코드를 검증한다. P2의 run-scoped overlay가 오면 이 함수는
  수정 없이 그대로 patched build를 검증하게 된다 — 코드 재작업 불필요, P2 작업만 기다리면 됨.
- **[가정] `check_static`의 기준선은 원본 source, 비교 대상은 patched worktree**: 둘 다
  `scanners.sast.run_semgrep` + `scanners.aggregate.aggregate`로 스캔해 high/critical 후보
  수를 비교한다. semgrep 미설치 환경에서는 `SemgrepUnavailableError`가 그대로 전파된다
  (`vc_run_sast`와 동일 제약, 알려진 한계로 문서화했을 뿐 오늘 새로 발견한 리스크는 아님).
- **[리스크] `26s-w1-c2-04` 실제 라이브 리허설 미완**: 이 세션 환경엔 P2의 실제 source clone이
  없어(gitignored, 로컬 전용) register+readiness 확인까지만 했다. 코드 배선은 mock 기반
  단위 테스트 43건 + `vc_apply_patch`의 실제 git 프로세스 검증으로 커버했지만, 실 컨테이너
  build/health/HTTP 공격을 묶은 end-to-end는 아직 실증되지 않았다 — P3가 자신의 환경에서
  이어가기로 사용자와 합의(위 "다른 역할에 필요한 사항 → P3" 참고).
