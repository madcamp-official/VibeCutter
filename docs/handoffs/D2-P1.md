# D2 / P1 Handoff

## 상태

완료 — `plan.md` Day2 섹션에 기록된 5개 블록(어제 구멍 메우기 / 정책 등록 / verify tool 실배선 /
judge.py skeleton + Attack gate 실동작 / findings resource 완성 / 공통 계약 이견 정리)을 전부
처리했다. 전체 회귀 71개 통과. **이 문서는 그날 저녁에 게시됐어야 했는데 빠져 있던 것을
Day3 완료 시점에 뒤늦게 채운다** — 내용은 당일 `plan.md` Day2 섹션에 실시간으로 기록해 둔
것을 그대로 정리한 것이라 새로 지어낸 내용은 없다.

## 변경 파일

- `core/evidence_store.py`: **구멍① 허구 evidence_id 승격 차단** — `update_finding_status()`가
  각 `evidence_id`를 `get(Observation, eid)`로 실존 확인 + `Observation.run_id == finding.run_id`
  일치 검사. `find_or_create_finding()` 신규(Candidate→Finding 지연 생성), severity/owasp 반영.
- `core/redaction.py`(신규): **구멍② secret redaction** — `write_artifact()` 저장 직전 적용.
  JWT(Bearer 유무 무관)/JSESSIONID/password 패턴 redact, hash는 redaction 후 bytes 기준.
- `mcp_server/tools_analysis.py`: **구멍③ max_requests 상한** — `vc_verify_*` 세 함수를
  `Annotated[int, Field(ge=1, le=20)]`로 변경. **verify tool 실배선** — `approved: bool` 승인
  게이트, `_prepare_verification()` 공통 헬퍼(승인→policy→VERIFYING 전이→Finding 지연 생성→
  verifier 호출→상태 갱신), `vc_verify_access_control` 본문 실제 구현, `vc_verify_injection`/
  `vc_verify_xss`는 verifier 직전까지만.
- `contracts/schemas.py`: `VerificationResult` 신설(`VerifyResult`/`VerifierOutput` 중복 제거),
  `ObservationType` StrEnum 신규(`http_exchange | db_diff | browser_trace | log | route_map | role_map`),
  `Candidate.vuln_class`/`attack_params` 추가(기존 `signals` 유지, additive), `Finding.affected_role`
  → `affected_roles: list[str]`로 리네임(사용처 0건 확인 후).
- `verifiers/types.py`: `VerifierOutput = VerificationResult` 별칭(P3 `verifiers/access_control.py`
  무수정으로 계속 동작).
- `policies/scope.yaml`: D1-P2.md가 제시한 실제 build/health 통과 target 19개의
  `allowed_hosts`/`port` 등록(3개는 의도적 제외, 이유 주석).
- `policies/commands.yaml`: `build_target`/`start_target`/`reset_target` command_id typed args 등록.
- `core/judge.py`(신규): 6개 게이트 함수 시그니처(`check_build/check_attack/check_positive_functionality/
  check_regression/check_static/check_scope`) 전부 정의, **Attack gate만 실제 동작** —
  `check_attack(run_id, finding_id, *, verifier=verify_access_control)`가 원본 candidate로
  verifier를 재호출해 `verified=False`를 판정. 나머지 5개는 `NotImplementedError` 스텁.
- `mcp_server/resources.py`: `vibecutter://findings/{finding_id}`가 `evidence_store.get(Finding, ...)`를
  실제로 조회하도록 변경, 더미 데이터/미사용 import 제거.
- `tests/test_evidence_store.py`, `tests/test_redaction.py`, `tests/test_verify_tool_wiring.py`,
  `tests/test_judge.py`, `tests/test_resources.py`, `tests/test_schema_contract_changes.py`: 위
  변경 전부에 대한 회귀 테스트(신규 33건).

## 제공 인터페이스

- `core.evidence_store.update_finding_status(finding_id, status, *, evidence_ids)`: 이제
  evidence_id 실존 + run 일치까지 검증(허구 id로 승격 불가).
- `core.redaction`: `write_artifact()`가 저장 전 자동 적용(호출부는 신경 쓸 필요 없음).
- `vc_verify_access_control(run_id, candidate_id, *, approved: bool = False, max_requests: int = 10)`:
  MCP 경로로 실제 `verifiers.access_control.verify()`를 호출해 Finding을 verified/rejected로 승격.
  `vc_verify_injection`/`vc_verify_xss`는 동일 게이트를 타지만 verifier 호출부는 미구현.
- `core.judge.check_attack(run_id, finding_id, *, max_requests=10, verifier=verify_access_control) -> bool`:
  유일하게 실동작하는 게이트. 나머지 5개는 호출 시 `NotImplementedError`.
- `vibecutter://findings/{finding_id}`: 실제 저장된 Finding을 그대로 반환(policies도 실데이터,
  targets/manifest/run/evidence는 아직 더미).
- `contracts.schemas.ObservationType`/`Candidate.vuln_class`/`attack_params`/`Finding.affected_roles`:
  전부 additive — 기존 `"http_exchange"` 문자열 리터럴, `signals` 파싱, `affected_role`(단수) 키워드는
  에러 없이 계속 동작(단, `affected_role` 키워드는 조용히 무시됨 — `extra="forbid"` 미설정이 한계).

## 검증

- 전체 회귀 테스트 71개 통과(`python -m unittest discover -s tests`, P1 신규 33건 포함).
- `core.policy_engine.require_target_allowed`/`require_host_allowed`/`require_valid_command`를
  실제 target_id(`26s-w1-c1-03`, `26s-w1-c2-07`)로 직접 호출해 통과 확인, 미등록 target은 여전히 거부.
- `vc_verify_access_control`을 실제 `mcp.call_tool()` 경로로 호출(verifier는 mock)해
  verified→Finding 승격, rejected→미승격, evidence 기록까지 확인.
- **21개 target 전체 round-trip은 P2 manifest 블로커로 실패** — `TargetCatalog.load()`가 3개
  manifest(`secret_env_names` 규칙 위반)에서 전체 실패. 정상 manifest 1개(`26s-w1-c1-03`)만
  담은 임시 catalog로 `register()`/`check_readiness()`까지는 실제 프로덕션 경로로 검증 완료.
- `.vibecutter/evidence.db`가 컬럼 추가 전 스키마로 이미 존재해 스키마 변경 후 19건 실패 →
  로컬 DB 삭제 후 재생성으로 해결(마이그레이션 도구 없음, 알려진 한계).

## 다른 역할에 필요한 사항

- **P2**: 🔴 3개 manifest(`26s-w1-c2-05`/`26s-w1-c2-08`/`26s-w1-c3-04`)의 `role_fixtures[].secret_env_names`가
  `VIBECUTTER_*` 규칙을 어겨 `TargetCatalog.load()` 전체가 실패한다 — 최우선 수정 요청.
  🟡 `runtime/target_service.py` build 실패 경로가 `target.id`(존재 안 함)를 참조해
  `AttributeError`가 남 — `target.manifest.id`로 수정 요청. `policies/scope.yaml`/`commands.yaml`
  등록 내용(host/port) 최종 확인 요청. IDOR 검증 가능한 target(사용자 2명 + 각자 소유 seed 자원)
  1개 지정 요청.
- **P3**: 구멍①②③ 수정 완료 + `VerifyResult`/`VerifierOutput`을 `contracts.schemas.VerificationResult`로
  통합 완료(별칭 유지, `verifiers/access_control.py` 무수정) 공유. 저장 계층 redaction과 verifier의
  임시 `redact()`가 중복 적용돼도 idempotent라 지금 당장 지울 필요 없음. `Candidate.vuln_class`/
  `attack_params` 신규 필드는 강제 이전 없이 병행.
- **P4**: `Observation.type`을 `ObservationType` enum으로 고정 완료(`http_exchange` 포함, 기존 코드
  영향 없음). batch scan JSONL 산출물을 evidence/candidate store가 흡수할 포맷 확정 필요.
- **전원**: 스키마 변경 시 로컬 `.vibecutter/evidence.db`를 지워야 한다(`SQLModel.create_all()`은
  기존 테이블에 컬럼을 추가하지 않음, 마이그레이션 도구 미도입).

## 결정·가정·리스크

- **Finding 상태 전이는 `verification_state=` grep으로 우회 경로 재확인** — `update_finding_status()`
  내부 1곳과 `resources.py` 더미 생성자(영속화 안 됨) 외엔 없음. 하드 가드 유지.
- **공통 계약 변경 3건(Observation.type/Candidate.vuln_class·attack_params/Finding.affected_roles)은
  전부 additive/무해** — 각 필드의 실제 사용처를 grep으로 먼저 확인한 뒤에만 변경(리네임한
  `affected_role`은 사용처 0건이라 마이그레이션 걱정 없이 진행).
- **오늘이 사실상 마지막 무료 스키마 변경 창구였다(P3 지적)** — 로컬 DB가 비어 있어 자유롭게
  바꿀 수 있었을 뿐, 실제 target으로 run이 쌓이기 시작하면 이후 스키마 변경은 전원이 로컬 DB를
  지우거나 마이그레이션이 필요해진다.
- **verify tool은 access_control만 실배선, injection/xss는 P3 verifier 미구현이라 스텁**:
  policy/상태 전이 배선은 동일하게 타지만 verifier 호출부에서 `NotImplementedError`.
- **21개 target 전체 실증은 P2 블로커로 미완** — subset(`26s-w1-c1-03`) 검증으로 정책 등록
  로직 자체엔 문제 없음을 확인했으나, catalog 전체 로드가 3개 manifest 때문에 죽어 있어 실제
  target으로 candidate를 만들 방법이 아직 없었다(Day3에 P2가 해소).
