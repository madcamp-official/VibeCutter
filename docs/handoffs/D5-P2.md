## 공통 계약 결정 — candidate 하나당 worker Run 하나

P2는 P1의 **candidate마다 새 Run** 제안을 지지한다. `VERIFIED`는 `LOCALIZING`으로만
진행하는 고정 상태이므로, 하나의 Run이 여러 verified candidate를 담는 컨테이너가 되면
상태 머신·patch·validation·evidence가 서로 섞인다. P2의 worktree와 generated Compose overlay도
이미 `target_id + run_id` 단위이므로 이 선택이 격리 및 reset 경계와 일치한다.

P1 batch/orchestrator가 구현할 최소 계약은 다음과 같다.

1. scan Run은 후보를 수집하는 batch 부모로만 사용하고 `CANDIDATE_SCAN`에서 종료한다.
2. 후보마다 별도 worker Run을 만들고, worker Run에 속한 Candidate를 materialize한 뒤 verify 한다.
   원본 scan candidate ID는 additive lineage 필드(권장: `origin_candidate_id`) 또는 동등한
   trajectory metadata로 보존한다. 기존 Candidate의 `run_id`를 덮어쓰지 않는다.
3. Finding, Observation, Patch, Validation, Trajectory는 모두 worker Run에만 저장한다.
   따라서 한 worker Run은 verified이면 patch loop로, rejected이면 종료로 독립 진행한다.
4. 같은 fixed host port를 쓰는 현재 P2 manifest는 worker Run을 **target별 순차 실행**해야 한다.
   병렬 worker runtime이 필요하면 P2에 port allocation 계약을 별도로 요청한다.

P3 verifier는 derived Candidate의 typed `attack_params`를 그대로 소비하면 된다. P3가
scan candidate ID를 report/dataset에서 추적해야 하면 위 lineage 값을 사용해야 하며, worker
Run의 evidence와 섞지 않는다.

## P2 runtime 호출 조건

- batch 시작 전: `sweep_stale_run_overlays(target_id, active_run_ids=..., approved=True)`.
  이 함수는 P2가 생성한 trusted overlay artifact만 정리하며 임의 `vc-*` project는 삭제하지 않는다.
- approved patch가 overlay를 생성한 worker Run 종료 `finally`: `reset_run(target_id, run_id,
  approved=True)`. reset 실패 시 artifact/worktree를 남겨 진단·재시도를 가능하게 한다.
- patch 전 base runtime은 target별로 공유할 수 있으나, write verifier 뒤에는 P2 target reset과
  fixture 재준비가 필요하다. 인증/상태변화 fixture는 P3가 target·safe mutation·원복 방법을
  명시해 요청할 때만 P2가 추가한다.

## 제공 인터페이스

- 입력: policy-allowed `target_id`, active worker `run_id` 집합, 명시 승인.
- 출력: `StaleRunSweepResult(cleaned_run_ids, failed_run_ids, skipped_active_run_ids)` 및
  run-scoped Compose/worktree cleanup 결과.
- 실패/예외: approval 누락, untrusted artifact path, reset 실패는 명확히 거부/보존한다.

## P2 다음 작업 계획

1. **source bootstrap 재현성:** source 전문은 계속 gitignore에 두되, clean host에서 target ID별
   repository/revision을 재현할 수 있는 bootstrap/lock 계약을 추가한다.
2. **GPU runtime preflight:** 7/7/6 queue의 checkout revision, source 존재, Docker service,
   고정 port 상태를 audit 시작 전에 확인한다. 원격 dispatcher는 P1/P3가 필요성을 요청할 때만 만든다.
3. **fixture 지원:** P3가 구체적인 role/resource/endpoint 계약을 주는 target부터 no-secret
   provisioning을 확장한다. 검증되지 않은 fixture를 15개 target에 일괄 추정해 넣지 않는다.
4. **write verifier 위생:** write 검증은 baseline 상태를 바꿀 수 있으므로 worker 종료 뒤 manifest
   reset과 fixture 재준비를 명시한다. 현재 driver의 `reset_run`은 patched overlay 전용이므로
   shared baseline reset은 별도 승인 단계로 유지한다.
5. **최종 통합 확인:** P3의 추가 run이 나오면 한 target에서
   `register → build/start → scan → worker verify → evidence/report → reset`을 재실행하고,
   포트·overlay·worktree 잔여 0을 확인한 뒤 P2 handoff를 종료한다.

## 결정·가정·리스크

- P2는 P1/P3 소유인 state machine, Candidate schema, verifier, orchestration 코드를 직접 변경하지
  않는다. 이 handoff는 해당 변경의 명시적 합의 입력이다.
- `reset_run`은 run-scoped patched overlay용이다. overlay가 없는 scan/verify worker에 무조건 호출하지
  말고, P1은 overlay 생성 여부를 기준으로 호출해야 한다.
- write verifier가 shared baseline의 데이터를 변경했다면 `reset_run`만으로는 원복되지 않는다.
  승인된 manifest reset과 fixture 재준비가 별도로 필요하다.
- 병렬 Compose worker는 현재 고정 host port와 충돌한다. 새 port allocation/URL contract 없이 병렬화하지
  않는다.
