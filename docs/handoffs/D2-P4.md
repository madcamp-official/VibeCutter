# D2 / P4 Handoff

## 상태

**GPU 불필요 항목 전부 완료. base LLM 추론 우선순위·모델 서빙만 GPU 대기.**

완료: ① candidate 통합·중복제거·FP reject·우선순위(`scanners/aggregate.py`),
② **RAG 우선순위**(`scanners/rag_enrich.py` — code_index 검색 신호를 우선순위에 반영),
③ trajectory 수집 인프라(`model/trajectory.py`), ④ B1/B2 baseline 실행 배선
(`eval/run_baseline.py`), + batch `--benchmark` 플래그.

보류: base LLM **추론** 재랭킹(휴리스틱+RAG 까지 완료, LLM 부분만 `rerank_fn` 훅=GPU), 모델 서빙(GPU).

**D2 완료 기준("verified 후보에 evidence 연결")은 P4 단독으로는 못 닫는다** — 후보 통합/
우선순위(P4)는 됐고, 실제 verified 판정·evidence 연결은 P1 judge + P3 verifier 소관.
내 쪽은 그 결과를 받을 준비(aggregate.kept → verify, TrajectoryRecorder)가 완료.

근거: 5일 계획 D2 P4 칸, 기획서 7.2절(FP reject)·4.5절(trajectory)·12.2절(baseline),
cowork_rule.md 5절(evidence 연결된 trajectory만 학습).

## 변경 파일

- `scanners/aggregate.py`: 여러 스캐너 Candidate[] → 병합(같은 위치·focus·cwe)·FP reject
  (비-앱 경로 기본, 저신뢰는 opt-in)·우선순위(confidence + 교차검증 + severity + **RAG 관련도**).
  `rerank_fn` 훅(LLM 재랭킹 자리, GPU). severity 어휘 SAST/SCA 모두 인식. (`_tool_of` dead code 제거)
- `scanners/rag_enrich.py`: **RAG 우선순위** — `code_index` 로 candidate 위치의 코드 컨텍스트를
  검색해 `rag:loc/relevance/symbols` signal 부착. focus별 sink 어휘 매칭으로 관련도(0~1) 계산.
  LLM 없이(GPU 없이) 도는 검색 신호 → `aggregate.priority_score` 가 최대 +0.1 반영.
- `model/code_index.py`(추가): `chunk_at(file, line)` — 위치→코드 chunk 조회(RAG 보강용).
- `model/trajectory.py`: `TrajectoryRecorder`(상태 전이 스텝을 JSONL 기록), `load_trajectories`,
  `training_samples`(evidence/validation 연결된 것만 — label∈{verified,fixed,rejected,
  human_review} 또는 reward 있는 스텝), `to_sft_sample`, `stats`.
- `eval/run_baseline.py`: 배치 산출 candidate JSONL → `{app_id: set[focus]}` 예측 → 벤치마크
  정답 대조 → BaselineReport. B1(Semgrep)/B2(ZAP) 공통.
- `scanners/batch_scan.py`: `--benchmark` 플래그(정답 있는 벤치마크 앱 스캔 → B1/B2 측정용).
- `scanners/vocab.py`: **severity 5단계**(critical/high/medium/low/info) + **OWASP 2021**
  카테고리 값 집합 + `candidate_severity/candidate_owasp`(candidate signal → Finding 필드
  매핑). **D1-P1.md "severity/owasp_category 합의된 값 집합 알려달라" 요청 응답.**
- `model/trajectory.py`(추가): `OBSERVATION_TYPES`(http_exchange|db_diff|browser_trace|
  log|route_map|role_map) 채택 + `valid_evidence`/`to_sft_sample(observations=)` 증거 조인.
  **D1-P3.md 이견 3 "Observation.type 값 집합 확정 — P4 trajectory 조인이 여기 걸린다" 응답.**
- 각 모듈 `test_*` + 신규 스위트(aggregate9/trajectory8/run_baseline3/vocab4/rag_enrich6). 전체 **66 pass**.

공통 계약(`contracts/schemas.py`, 상태 이름, manifest)은 변경하지 않음 — 소비만 함.
다른 역할 소유 파일(mcp_server/, core/, runtime/, scanners/playwright.py 등) 미변경.

## 제공 인터페이스

### 1. Candidate 통합 (`scanners/aggregate.py`)
- `aggregate(*candidate_lists, min_confidence=0.0, rerank_fn=None) → AggregateResult`.
  `AggregateResult.kept`(우선순위 내림차순 Candidate[]), `.rejected`([(Candidate, 이유)]),
  `.summary`(kept/rejected/by_focus/reject_reasons).
- `merge_duplicates(cands)`, `priority_score(c)` 도 개별 노출.
- **P3 는 raw candidate 대신 `kept` 를 verify 하면 됨**(FP·중복 이미 제거).

### 2. Trajectory 수집 (`model/trajectory.py`)
- `TrajectoryRecorder(path).record_step(run_id, state, action, result, next_state,
  reward=None, label=None) → Trajectory`(JSONL append).
- `load_trajectories(path)`, `training_samples(trajs, require_label=True)`,
  `to_sft_sample(traj)`, `stats(trajs)`.
- 입력/출력: `contracts.schemas.Trajectory`. label 은 P1 judge 판정(verified/fixed 등)을 받아 채움.

### 3. Baseline 실행 (`eval/run_baseline.py`)
- `predictions_from_candidate_dir(dir) → {app_id: set[focus]}`(`<app>.candidates.jsonl` 읽음).
- `baseline_report(dir, ground_truth) → BaselineReport`.
- CLI: `python -m eval.run_baseline --candidates <dir> --label B1`(정답=벤치마크 inventory).

## 다른 역할에 필요한 사항

- **P3 (최우선)**: **SAST 가 이제 `candidate.vuln_class`(idor/xss/injection)를 채운다 →
  verifier dispatch 를 `signals` 파싱 대신 `candidate.vuln_class` 로 바꿔도 된다**(D1-P3 이견1·
  D2-P3 우회 제거 조건 충족). 그 외:
  - verify 대상은 raw 목록 말고 **`scanners.aggregate.aggregate(...).kept`**(중복·FP 제거·우선순위순).
    signal 규약: `focus:<group>`, `category:sca`, `severity:<...>`, `rag:relevance=<0~1>`.
  - `Observation.type` 은 이제 `contracts.ObservationType` enum(P1 채택). validation 을
    `http_exchange`(producer `vc_validate_patch:...`)로 저장하면 내 trajectory 조인에 그대로 붙음 — 이미 그렇게 하는 중, 유지.
  - **API 유지 약속**: `CodeIndex.build(root).search(q,k)`·SAST `source_symbols("파일:줄")` 포맷 안 바꾼다(P3 소비 확인).
- **P1**: (a) **trajectory 기록 주체 결정** — 상태 전이 시 P1 이 `model.trajectory.TrajectoryRecorder.record_step(...)`
  호출 vs P4 가 evidence_store 에서 사후 조립. (b) aggregation(`aggregate.kept`)을 CANDIDATE_SCAN→VERIFYING
  사이 어디에 끼울지. (c) **D3 리포트 조인 데이터**(finding+evidence+patch+validation) 형태를 주면
  HTML/SARIF export 를 맞춘다. (d) **severity/owasp 값 집합**(`scanners/vocab.py`:
  `Finding.severity`∈{critical,high,medium,low,info}, `owasp_category`∈OWASP 2021) 스키마/문서
  채택 여부. (e) `vc_run_sast`↔`scanners.sast.run_semgrep`, `vc_run_sca`↔`scanners.sca.run_osv` 배선.
- **P2**: **batch_scan 이 `--use-p2-sources`(`.vibecutter/targets/sources/<id>`)로 재-clone 없이
  스캔하도록 반영 완료** → 그 경로가 맞는지 + 소스 clone 준비 시점 확인 부탁. inventory↔manifest
  역할 분리·adapter override(`26s-w1-c1-01`,`26s-w1-c3-05`) 합의는 반영 완료(ack).
- **전원**: (1) `requirements.txt` 에 `semgrep==1.90.0` 재추가함(merge 에서 빠졌던 것 — 공유 파일이라
  머지 시 인지). (2) **Python 버전 통일 필요** — README 3.11 vs 실제 3.12/3.14 혼재(P3 D1 제기).
  P4 코드는 3.11~3.14 전부 동작하나 팀이 하나로 정하는 게 좋다.

## 결정·가정·리스크

- **[결정] FP reject 는 보수적**: 기본은 비-앱 경로(tests/vendor/dist/생성물)만 제거. 저신뢰
  제거는 `min_confidence` opt-in — 실제 취약점을 날리지 않기 위함. 교차검증된 후보는 저신뢰라도 유지.
- **[결정] 우선순위 severity 어휘 통합**: SAST(ERROR/WARNING/INFO)·SCA(CRITICAL/HIGH/…) 둘
  다 인식하도록 수정(초기엔 SCA 어휘만 인식해 코드흐름 후보가 부당하게 밀림 — 수정).
- **[가정] trajectory 학습 필터**: label∈{verified,fixed,rejected,human_review} 또는 reward
  있는 스텝만 학습에 사용(cowork_rule 5절 "evidence·validation 연결된 것만"). 실제 label 은
  P1 judge 가 채운다.
- **[훅] LLM+RAG 재랭킹은 `rerank_fn`**(base LLM 추론은 GPU) — D4 모델 서빙 후 연결.
- **[리스크 이월] requirements.txt `semgrep`**: 워킹트리엔 재추가함(`semgrep==1.90.0`). 공유
  파일이라 커밋 시 머지 조율 필요. osv-scanner 는 Go 바이너리(외부 설치).
- **[리스크 이월] `scanners/` 공유**: 커밋 시 내 파일만 add(P3 `playwright.py` 오염 방지).

## 통합(rebase) 반영 — 최신 main 계약 변화 대응

rebase 로 P1 이 P3·P4 요청을 **공통 계약에 정식 채택**했다. 그에 맞춰 P4 코드를 정렬했다.

- **`Observation.type` → `contracts.ObservationType` enum 채택** (내 D1-P3 응답이 계약이 됨).
  값 동일(`http_exchange|db_diff|browser_trace|log|route_map|role_map`). `model/trajectory.py`
  의 `OBSERVATION_TYPES` 를 **enum 에서 파생**하도록 바꿔 drift 차단. 잘못된 type 은 이제
  스키마가 생성 단계에서 거부(테스트도 그에 맞게 수정). **P3 가 validation evidence 를
  `http_exchange`(producer `vc_validate_patch:post_patch_{baseline,attack}`)로 저장** → 내
  trajectory/리포트 조인에 그대로 붙는다(D3-P3 확인).
- **`Candidate.vuln_class` / `attack_params` 필드 추가** (P3 이견1 채택). SAST 가 이제
  `vuln_class=focus`(idor/xss/injection)를 채운다 → **P3 verifier 가 `candidate.vuln_class`
  로 검증 모듈을 분기**(verifiers/types.py). `attack_params` 는 P3 소관(SAST 는 안 채움).
- **batch_scan P2 소스 재사용**: P2 가 알려준 경로(`.vibecutter/targets/sources/<id>` /
  `TargetCatalog.source_root_for`)를 `--use-p2-sources`(=`source_root_fn`)로 스캔 →
  **재-clone 방지**(D1/D2 조율 항목 해소). 소스 없으면 `source_missing` 로 기록.
- **inventory ↔ P2 manifest 역할 분리 합의 반영**: `datasets/inventory.yaml`=41 discovery/
  coverage catalog, P2 manifest=22 executable runtime. adapter 차이는 P2 의
  `targets/inventory_adapter_overrides.yaml`(26s-w1-c1-01 fastapi→generic-docker,
  26s-w1-c3-05 fastapi→node)에 기록됨. inventory adapter 는 static 제안, runtime verdict 는 P2.
- **API 안정 약속(P3 요청)**: P3 locator 가 `CodeIndex.build(root).search(q, k=1)` 와 SAST
  `source_symbols` "파일:줄" 포맷을 소비 중 → 두 인터페이스 유지한다.

검증: 전체 **70 pass**(sast10/sca9/batch6/aggregate9/vocab4/rag6/baseline6/run_baseline3/
code_index8/trajectory9). rebase 후 계약 변화로 깨졌던 trajectory 테스트를 enum 기준으로 고쳐 복구.

**남은 워치 항목**: `attack_params` 는 P3 가 아직 signals 파싱으로 우회 중(D3-P3), 점진 마이그레이션.

## D2 GPU-보류 항목 완료 (GPU 서버 확보 대비, D4 시점 갱신)

D2 에서 GPU 대기로 남겼던 **모델 서빙 + `rerank_fn`/`embed_fn` 훅 연결**(위 §결정 [훅]
line 110, §상태 보류)을 GPU 없이 작성/테스트 가능한 형태로 **준비 완료**했다. GPU 서버가
나와서, 접근 확보되는 즉시 실서빙에 붙도록 뼈대를 미리 세운 것.
(QLoRA 학습은 D4 항목이라 여기 아님 — `model/train_lora.py` 는 D4-P4 로 분리 기록 예정.)

### 추가 파일
- `model/serving.py`: 서빙된 LLM 을 두 훅에 연결.
  - `make_rerank_fn(chat_fn)` → `aggregate(rerank_fn=)` 자리(D2 line 110 의 그 훅). LLM 이
    candidate 를 재정렬. **실패 시 입력 순서 유지(비파괴)** → aggregate 휴리스틱 정렬 보존.
  - `make_embed_fn(embed_call)` → `code_index.search(embed_fn=)` 자리. 실패 시 BM25 폴백.
  - `openai_chat_fn`/`openai_embed_call`/`health_check` — vLLM OpenAI 호환 `/v1/*` wrapper.
    **HTTP 는 stdlib(urllib)만** 써서 공통 requirements 에 의존성 추가 0.
  - 순수 파서 `_parse_rerank_order`(모델이 헛소리해도 항상 유효 순열 보장).
- `model/test_serving.py`: 목 chat_fn/embed_call 로 8/8 통과(네트워크/GPU 불필요).
- `requirements-gpu.txt`: 서빙/학습 의존성(vllm/torch/…)을 **공통 requirements.txt 와 분리** —
  GPU 서버에서만 설치(P1/P2/P3·Windows 머신 오염 방지).
- `docs/P4_MODEL_SERVING_RUNBOOK.md`: 서빙 기동→훅 연결→(D4)학습→평가 순서. P2 소유
  `P2_TARGET_RUNTIME_RUNBOOK` 컨벤션에 맞춘 P4 소유 런북(통합 RUNBOOK.md 의 P4 섹션).

### Python 3.13 팀 통일 (오늘 인프라 결정)
- semgrep 1.90.0 이 **3.11~3.13 정상, 3.14만 실행 불가**(opentelemetry)임을 WSL/Linux 에서
  실측 — 내 `run_semgrep` 파이프라인 + 유닛테스트가 3.12.3·3.13.14 둘 다 통과.
  이전 handoff(plan.md, D3/D4-P3, P2 status)의 "3.11/3.12 통일" 권고는 3.13 을 3.14 와
  뭉뚱그린 과잉 보수였다 → **3.13 으로 통일**(`.python-version` 이미 3.13, README 정정).
- Windows 팀원은 semgrep 만 WSL/Docker(OS 문제, 버전 무관). **SAST 실주행은 P4 가 WSL/3.13.**

### 이 항목으로 다른 역할에 필요한 사항
- **P2(또는 GPU 담당)**: GPU 서버 **접속 정보 + `nvidia-smi`** 를 P4 에 전달해달라
  (`P2_TARGET_RUNTIME_RUNBOOK` §GPU 의 model-serving 선행 조건). VRAM 으로 7B/양자화 확정
  (현재 24GB 가정: Qwen2.5-Coder-7B-Instruct, 4bit).
- **P1**: rerank_fn 을 실제 배선하는 지점(vc_run_sast/sca 이후 planner)에서
  `make_rerank_fn(openai_chat_fn(...))` 을 주입하면 kept 가 LLM 재정렬된다. endpoint 없으면
  주입하지 말 것(None 이면 휴리스틱 정렬 유지 — 비파괴).
- **전원**: `requirements-gpu.txt` 는 GPU 서버에서만 설치. 일반 개발 머신엔 깔지 말 것.
