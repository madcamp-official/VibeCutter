# D2 / P2 Handoff

## 상태
진행 중 — D2의 P3 patcher/validator와 P4 batch scan 통합을 확인했다. P2는 D1에서 target source Git worktree와 P3 IDOR fixture를 제공했고, D2에서 **run-scoped target worktree를 실제 Docker build/start context로 쓰는 Compose overlay**를 구현·실증했다. 사용자 결정에 따라 신규 target Dockerize 확장은 여기서 중단하고, 기존 20개 verified runtime을 학습/closed-loop 대상으로 사용한다. 원본 source clone과 원본 branch는 변경하지 않는다.

## 변경 파일
- `runtime/run_overlay.py`: checked-in manifest/Compose를 수정하지 않고 target Git worktree를 source build context로 바꾸는 run-scoped generated Compose 생성, static isolation 재검사, fixed lifecycle argv의 `-f` 경로 치환 실행기.
- `runtime/catalog.py`: `run_overlay_for(target_id, run_id)`로 target source repository/worktree에 묶인 overlay를 노출.
- `runtime/compose_isolation.py`: trusted generated Compose path를 같은 static isolation 규칙으로 검사할 수 있는 optional path 입력.
- `tests/test_run_overlay.py`, `tests/test_catalog.py`: worktree context rewrite, Dockerfile/Nginx asset path 보존, isolation 및 catalog overlay lookup 회귀 테스트.
- `targets/README.md`: P1 approval apply → P2 overlay build/start/health 절차와 고정 port/reset 조건 문서화.
- `docs/handoffs/D2-P2.md`: D2 P2 결과 및 후속 의존성 기록.

## 제공 인터페이스
- 입력: trusted `target_id`, valid `run_id`, P1 approval gate가 만든 detached target worktree.
- 출력: `catalog.run_overlay_for(target_id, run_id).prepare()` → `.vibecutter/run-overlays/<target_id>/<run_id>/compose.yaml`; `execute(command_id)` → `CommandResult`; `check_health()` → `HealthResult`.
- 실패/예외: source worktree 부재/target artifact root 이탈/non-Git path, untracked/unsafe compose path, loopback·egress isolation 위반, manifest command가 configured Compose file을 참조하지 않는 경우는 거부. approval 없는 patch/reset은 계속 거부.

## 검증
- D1 baseline: `26s-w1-c2-04` target source Git worktree create → `RunScopedTestRunner.run()` → approved remove 확인. test suite가 없어 verdict는 의도대로 `not_configured`.
- D1 baseline: `26s-w1-c2-04` API/UI healthy 상태에서 `prepare_idor_fixture` 실행, user A/B 및 baseline/attack metadata 생성 확인.
- D1 baseline: 전체 Python test suite 82 passed.
- D2: temporary Git source fixture에서 generated Compose의 build context가 detached worktree를 가리키고 Dockerfile/Nginx bind mount가 trusted P2 asset path를 유지하며 static isolation이 통과하는 단위 테스트 PASS.
- D2 actual: `26s-w1-c2-04` original runtime을 approved `down --volumes`로 내린 뒤, `.vibecutter/worktrees/26s-w1-c2-04/d2-c2-overlay` detached worktree 기반 generated Compose를 Docker `config --quiet` → build → `up -d --wait`로 통과. API/UI 모두 healthy (`127.0.0.1:14017`, `:14018`), `prepare_idor_fixture` 재실행 및 API health 200 PASS. 이 run instance는 P3 live verifier를 위해 유지 중.
- D2 closed-loop target feasibility: `26s-w1-c1-05`는 JWT 인증 Spring app이며 signup/login, owner/outsider team/task 모델, source regression을 갖춰 patcher demo 후보임을 확인했다. 다만 current P3 `IdorProbe.auth_mode`는 `none`/`session_form`만 지원하고 이 target은 JSON login → Bearer JWT 방식이다. P2는 role fixture의 secret/token을 파일에 저장하지 않으므로 P3 `bearer_json` replay contract가 먼저 필요하다.
- D2 예정: P1 approval apply가 실제 patch diff를 worktree에 적용한 뒤 동일 overlay로 patched build/start → P3 replay → run test suite를 한 번 연결.

## 다른 역할에 필요한 사항
- P1: `vc_apply_patch`가 P2 target worktree에만 diff를 적용하고 승인 뒤 `target_id`/`run_id`를 전달할 것. `26s-w1-c3-09` policy allowlist(`127.0.0.1:14036`)도 아직 필요.
- P3: `c2-04` fixture 기반 `auth_mode=none` verifier는 준비됐고 P2 D2 run instance가 `127.0.0.1:14017`에서 healthy 상태다. `.vibecutter/fixtures/26s-w1-c2-04-idor.json`을 읽어 live verifier를 실행할 수 있다. **다음 요청:** `c1-05` closed-loop fixture를 소비할 `bearer_json` auth mode(JSON signup/login response의 access token을 Authorization Bearer로 쓰는 baseline/attack GET)와 secret environment-name 입력 계약을 추가할 것. P2는 그 계약이 확정되면 local ephemeral owner/outsider + team/task fixture를 제공한다. patched live loop 대상은 P1 승인 patch가 필요하며, c2-04는 patcher 데모 대상이 아니므로 인증 Spring target을 별도로 쓴다.
- P4: `--use-p2-sources` source root 계약과 inventory/runtime adapter override 계약을 유지할 것.

## 결정·가정·리스크
- P2 run overlay는 static target manifest를 직접 수정하지 않고, run ID에 귀속된 generated configuration으로 source build context만 detached worktree로 전환한다.
- P1 policy가 고정 loopback port를 승인하므로, 첫 D2 리허설은 새 외부 port를 임의로 열지 않고 기존 target runtime을 approved stop/reset한 뒤 같은 allowlisted port에서 run instance를 기동한다.
- 생성 Compose가 source context만 바꿔도 Dockerfile·Nginx mount 등의 상대 경로 의미가 유지되는지 c2-04에서 검증했다. 다른 target의 build context/volume shape는 다를 수 있으므로 actual patch 대상마다 같은 static/Docker config 검증이 필요하다. 이 검증 전에는 closed-loop `fixed`를 주장하지 않는다.
- P2는 token/password를 manifest, fixture JSON, handoff, evidence에 저장하지 않는다. JWT target의 P3 replay는 P2가 지정한 `VIBECUTTER_*` role credential environment names를 process 환경에서만 해석하는 verifier contract가 필요하다.
