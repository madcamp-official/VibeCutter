# D4 / P2 Handoff

## 상태

진행 중 — target별 후보 선별·검증은 P3가 전담하고, P2는 runtime/provisioning/fixture/worktree/overlay/reset/test-runner
지원 역할을 수행한다. 2026-07-19 D4 runtime 위생 요청(MySQL readiness와 stale run overlay)을 구현·실측했다.

최신 main `07c9452`(P3 closed-loop/XSS·Injection, P4 3.13·WSL 공지 포함)을 P2 브랜치에 통합했다.
P2의 source-root/worktree 계약과 main의 테스트 하네스가 충돌한 부분은 해소했으며, 아래 작업 목록을
현재 P2 실행 기준으로 갱신한다.

## D4 runtime 위생 완료 항목

1. **c1-05 MySQL readiness** — `mysqladmin ping` 대신 앱과 같은 `MYSQL_USER`/`MYSQL_PASSWORD`로
   `MYSQL_DATABASE`에 `SELECT 1`을 수행하는 Compose healthcheck로 교체했다. mysqld 포트 오픈이 아니라
   init/grants 완료를 readiness 기준으로 삼는다.
2. **run-scoped cleanup** — `RunComposeOverlay.project_name`을 명시적으로 노출하고,
   `reset_run()`이 Compose teardown 성공 뒤 worktree가 이미 없는 경우에도 정상 종료되게 보강했다.
   reset 성공 시 해당 generated Compose artifact도 함께 제거한다.
3. **stale sweep** — `TargetRuntimeService.sweep_stale_run_overlays(target_id, active_run_ids=(), approved=True)`를
   추가했다. target의 generated artifact 아래에 있는 inactive run만 같은 `reset_run` approval 경로로
   정리하고, 실행 중 run과 임의 `vc-*` Docker project는 건드리지 않는다.
4. **c1-05 실측** — local ephemeral 환경값으로 fresh volume 3회에서 DB/app/frontend healthy와
   `GET /api/health = 200`을 확인했다. 별도 worktree `p2-c1-05-overlay-live`의 generated Compose도
   build → start → health 200 → `reset_run=True`을 통과했고, 종료 뒤 worktree·Compose project·generated
   artifact·14006/14007 포트가 모두 사라짐을 확인했다.
5. **실제 stale cleanup** — DB에 active run 기록이 없던 `d2-c2-overlay`(c2-04) 및
   `d4-c3-overlay-smoke`(c3-09)를 sweep으로 teardown했다. 남은 `vc-*` project는 없다.
6. **c2-02/c1-06 onboarding** — 두 source/manifest/Compose isolation이 로컬에 있으며, temporary local
   role env로 readiness `ready=True`을 확인했다. c1-06도 no-secret `self_signup`/`bearer` provisioning으로
   등록했다. P3 hint는 c2-02=`/api/auth/signup`, `accessToken`, `{"username":"{username}","password":"{password}"}`;
   c1-06=`/api/auth/signup`, `token`, `{"email":"{email}","password":"{password}","nickname":"{name}"}`이다.

## 다음 착수 조건

- **P1 batch/orchestration:** 매 run 종료에는 `reset_run(target_id, run_id, approved=True)`, batch 시작에는
  active run ID를 전달한 `sweep_stale_run_overlays(...)`를 호출해야 port leak가 자동으로 재발하지 않는다.
- **P1/P3 patched run:** 승인된 `run_id`/patch를 전달하면 P2는 같은 overlay 경로로 patched base URL을
  제공한다. P2는 verifier 재공격·`FIXED` 판정을 수행하지 않는다.
- **P3 fixture 요청:** 추가 write-IDOR은 target·safe `method/path/body`·`observe_path`·원복 방법이 지정된
  경우에만 no-secret fixture를 추가한다. 현재 write verifier는 무인증만 지원하므로 인증 write seed는 보류한다.
- **P4 Semgrep:** 팀 실행 기준은 Python 3.13 + `setuptools<81`이고 static scan은 P4의 WSL/Linux 환경에서
  수행한다. Windows native Semgrep은 P2 runtime gate가 아니다.

## 제공 인터페이스

- 입력:
  - policy-allowed `target_id`
  - P3가 요청한 fixture/provisioning schema
  - 승인 필요한 mutation 단계 여부(reset, fixture prepare, patched runtime start 등)
- 출력:
  - manifest 기반 build/start/readiness 결과
  - `vc_get_verifier_provisioning(target_id)` metadata
  - 승인된 no-secret fixture artifact
  - run-scoped worktree/Compose overlay
  - reset/rollback/test-runner 결과
- 실패/예외:
  - 임의 URL/IP/shell은 받지 않는다.
  - token/password/secret은 fixture/evidence/handoff에 저장하지 않는다.
  - reset, fixture prepare, patch apply, destructive test는 명시 승인 없이 실행하지 않는다.

## 검증

- 실행한 테스트와 결과:
  - 이후 사용자 지시로 Python 3.13 환경에 `requirements.txt`와 `pytest`를 설치했고,
    P2/P1 접점 테스트 `tests/test_catalog.py tests/test_apply_patch.py tests/test_fixture_contract.py
    tests/test_write_oracle_wiring.py tests/test_judge.py -q`가 **55건 PASS**했다.
  - 이번 P2 runtime 회귀 `tests/test_target_service.py tests/test_run_overlay.py
    tests/test_c1_05_runtime_contract.py tests/test_verifier_provisioning.py -q`가 **19건 PASS**했다.
  - c1-05 fresh-volume 3회 및 run-scoped overlay lifecycle을 실제 Docker Desktop Linux backend에서 확인했다
    (위 완료 항목 4 참조).
  - 전체 `pytest -q`는 P2 테스트 통과 뒤 P4 RAG의 기존 3건(`scanners/test_rag_enrich.py`) 실패를 보였다.
    P2 변경 파일과 무관하며 수정하지 않았다.
  - 전체 `pytest -q`에서 P2/P1 접점인 source-native `check_build`가 worktree repo root를 쓰는
    latent bug를 확인해 `run_source_root_for()` 기준으로 수정했다.
- 아직 실제 환경에서 검증하지 못한 항목:
  - 최신 main의 P1/P3 write-IDOR path와 P2 fixture 확장 간 end-to-end 검증은 P3가 target을 지정한 뒤 수행한다.
  - 추가 `safe_mutation` target은 P3가 실제 write candidate와 safe mutation 계약을 지정한 뒤에만 만든다.

## 다른 역할에 필요한 사항

- P3: P2 지원이 필요한 target마다 `target_id`, 필요한 인증/seed/fixture mode, role/resource/endpoint schema를
  구체적으로 요청해 달라. c2-02/c1-06에는 위 no-secret signup/token hint를 `_SELF_SIGNUP_HINTS` 또는
  `candidates_for_target(..., self_signup_hints=...)`로 연결하면 된다.
- P1: batch/orchestration 종료 시 `reset_run`을, 시작 시 `sweep_stale_run_overlays`를 배선해 달라.
  P2는 P1 judge가 호출하는 worktree/overlay/reset/test-runner 인터페이스만 유지한다.
- P4: 학습/평가 데이터는 P3/P1이 공통 evidence store와 judge를 통해 확정한 trajectory만 사용한다.
  폐지된 target 분담표의 “P2 실행 완료/blocked” 문구를 라벨로 사용하지 않는다.

## 결정·가정·리스크

- P2가 기존에 수행한 `c2-01`, `c2-02`, `c1-06`, `c1-07`, `c1-03` build/scan/replay 기록은 과거 실행
  이력으로만 남긴다. 이후 audit 완료 여부는 P3 실행 결과와 P1 judge로 판단한다.
- P2의 다음 독립 작업은 새 target 공격이 아니라 P3가 막힌 target의 runtime/fixture contract를 푸는 것이다.
- 최신 main 기준 write-IDOR 계약에 문서/코드 불일치가 남아 있다. D4-P3 handoff는 write candidate에
  `mutation_marker`를 담지 않는다고 설명했던 초안이 있었으나, 최신 `D4-P3.md`의 정정 상자와 현재 P1
  코드는 candidate에 `mutation_marker`/`extra_body_json`을 담는 것을 정본으로 확정했다. P2는 이 계약을
  따르되, 새 fixture의 인증 write 확장은 P1/P3 스키마 확정 전 임의로 추가하지 않는다.
- 로컬 기본 Python은 3.13으로 맞췄다. 기존 `tests/test_apply_patch.py`의 fake catalog는
  `run_source_root_for()`를 제공하지 않아 최신 P2 source-root apply 계약을 검증하지 못했으므로,
  테스트 하네스만 최소 수정했다.
- source-native patched build도 같은 계약을 따라 `run_source_root_for()`에서 실행해야 한다.
  현재 checked-in target은 대부분 Compose overlay 경로를 타지만, non-Compose target이 추가되면
  이 경로가 실제 build 기준이 된다.
- sweep은 managed artifact가 있는 run만 정리한다. artifact 없이 남은 임의 `vc-*` project를 이름만으로
  지우면 active run을 오판할 수 있으므로 P1이 active run registry를 제공하기 전에는 삭제하지 않는다.
