# D4 / P2 Handoff

## 상태
진행 중 — D4 clean-room/rollback 준비를 시작했다. P3 live verifier 대상 `26s-w1-c2-04`는 유지하고,
secret 없는 reset/smoke runtime인 `26s-w1-c3-09`를 holdout/demo clean-room 후보로 준비했다.

## 변경 파일
- `docs/P2_TARGET_RUNTIME_RUNBOOK.md`: clean-room 대상 구분, explicit reset, patched run rollback 절차를 기록.
- `docs/handoffs/D4-P2.md`: D4 P2 준비 상태와 다른 역할의 연결점을 기록.

## 제공 인터페이스
- 입력: policy-allowed `target_id`, explicit approval, 그리고 patched run이면 trusted `run_id`.
- 출력: static preflight/readiness 결과, manifest-defined build/start/reset/smoke 실행 경로,
  `TargetRuntimeService.reset_run()`의 성공 여부.
- 실패/예외: approval 없는 reset, policy 밖 target, generated Compose reset 실패는 거부 또는 실패로
  처리하며 실패한 run worktree는 삭제하지 않는다.

## 검증
- `26s-w1-c3-09`: `docker compose config --quiet` PASS, 현재 Compose service 없음 확인,
  `catalog.readiness_for()` → `ready=True`, unavailable executable 없음.
- 실제 reset/build/start/smoke는 아직 수행하지 않았다. 이들은 destructive lifecycle 단계이므로
  P1/P3의 live run과 충돌하지 않는 명시 승인 뒤에만 한다.

## 다른 역할에 필요한 사항
- P1: Day4 kill switch/rollback은 P2 `reset_run(target_id, run_id, approved=True)`를 호출해
  generated Compose reset 성공 뒤에만 worktree를 정리하도록 배선할 것.
- P3: `c2-04` live verifier가 끝났다는 신호 전까지 P2가 해당 target reset을 수행하지 않는다.
- P4: holdout `c3-09`은 runtime clean-room 후보일 뿐, 평가 데이터 편입 전에는 P3 evidence와
  P1 deterministic verdict가 필요하다.

## 결정·가정·리스크
- `c3-09` 선택은 loopback Compose, local MySQL volume reset, source-local seed/smoke 기준이다.
  보안 취약점 여부나 최종 데모 사용 승인은 의미하지 않는다.
- P2는 VM snapshot을 임의로 만들지 않는다. 현재 portable snapshot 동등물은 manifest의 approved
  `down --volumes` reset이며, 실제 VM snapshot authority가 필요하면 사용자 지시가 필요하다.
